## 二、数据与表约定

### 2.1 当前任务（ProductionOrder）与任务明细（ProductionOrderDetail）写入时机

- **致美斋项目参考**：当前项目致美斋部分在**添加任务**时写入与更新 ProductionOrder / ProductionOrderDetail。
  - **写入**：用户通过任务管理「添加任务」调用 `POST /api/task/add` → `TaskApplicationService.addTask` → `TaskRepositoryImpl.save(task)`。在 `TaskRepositoryImpl.save` 中：**先** insert **ProductionOrder**（生成 OrderNo，设置 Type、OrderStatus、OrderSumCount、ProductSumCount、DmakeDate、ProductSourceType=2、ProductionLine 等），**再** insert **ProductionOrderDetail**（关联 OrderId、OrderNo，以及产品、计划数量、生产时间等）。见 [TaskRepositoryImpl.save](miduo-variant-zhimeizhai/src/main/java/com/miduo/cloud/variant/zhimeizhai/persistence/repository/TaskRepositoryImpl.java)（约 35～58 行）。
  - **更新**：启用/停用任务时 `updateTaskStatus` 更新 ProductionOrder 的 OrderStatus（及 ProductionOrderDetail 关联状态）；成垛/采集完成时在 `CodeApplicationService` 等中更新 ProductionOrderDetail 的 OrderCount、ProductionOrder 的 OrderSumCount。
- **石湾 P02-01 约定**：石湾**无**“先任务管理添加任务再选单”的流程；当前任务在**开始采集门禁通过后**由 2 号机后端写入。后端提供“写当前任务”接口，在门禁通过后根据前端传入的订单号、产品编号、产品名称、采集规格（M、N）**生成或更新** ProductionOrder 与 ProductionOrderDetail（顺序与字段设计参考致美斋先 ProductionOrder、后 ProductionOrderDetail），并返回当前任务标识供后续盒箱关联等使用。

### 2.2 剔除记录表（需补充至数据库设计文档）

- 技术分析：执行剔除时往**剔除记录表**写入一条记录，字段含：箱码、盒码、瓶码明细、问题码、剔除原因、剔除时间、生产单号等。
- 建议在 [数据库设计.md](docs/石湾开发文档/数据库设计.md) 中新增表（表名与字段命名 PascalCase，与现有风格一致），例如 `RejectRecord`：Id、OrderNo、CaseCode（箱码）、BoxCode（盒码）、BottleCodesDetail（瓶码明细或 JSON）、ProblemCode、RejectReason、RejectTime、CreateTime 等；并建索引 OrderNo、RejectTime，便于总剔除数按订单/任务统计与 P02-07 剔除记录弹窗查询。

### 2.3 采集规格持久化

- 需求：采集规格「1垛 [70]箱」「1箱 [4]盒」读取本地文件；若已有设置则开始采集时先判断，有则弹变更提示，无则直接保存到本地文件。
- 实施：前端或后端在用户点击「开始采集」且门禁通过前，读取本地配置文件（或后端提供的“当前产线规格”接口）；若本次界面上的 M、N 与已保存不一致则弹窗提示是否覆盖；确认开始后写回本地文件（或调用后端保存），开始采集后输入框只读。

## 三、产品选择与生产单号

### 3.1 产品选择弹窗

- **需求**：点击主界面**产品选择下拉框**弹出产品选择弹窗；弹窗内搜索、产品表格（产品名称、产品编号）、分页；选中一条确认后关闭，主页面回显产品名称、产品编号（只读）；开始生产后不可更改。
- **接口说明**（详见 [技术分析 PM2603 310～450 行](docs/石湾开发文档/赋码采集关联系统T1.2.0（石湾版）技术分析PM2603.md)）：
  - **方法**：POST  
  - **地址**：开放平台 path 为 `/api/sign/md.shop.products/v1/list`；**完整 URL = base URL（IP/域名）+ path**。**base URL 与签名**由 [2 号机系统设置](.cursor/plans/2号机系统设置补充实现_1d319ad3.plan.md) 的**接口配置**持久化（shiwan-m2-settings.json），启动时加载，**不写死在代码中**。  
  - **请求参数**：page（必填，从第几页）、pagesize（必填，条数）、query（必填，JSON 查询条件，子项如 prostate、name、category、pronumber 等，值为空则不查该条件）、sort（可选，排序）。  
  - **返回**：error_code、error_msg、return_data.total、return_data.datalist[]；datalist 单项含 productid、name、pronumber、status 等，表格展示时取 **name**（产品名称）、**pronumber**（产品编号）。  
  - 可由 2 号机后端代理该接口（读取配置中的 base URL 与签名后请求开放平台），前端调 `POST /api/shiwan/products` 或等价。
- **样式与效果**：参考 [docs/前端界面实现可参考内容.md](docs/前端界面实现可参考内容.md)（base-styles、theme-colors、button-styles、dialog、表格与分页等）；实现效果以 [设计图/选择产品.png](.requirement/V1.2石湾产线采集关联/设计图/选择产品.png) 为准：标题「选择产品」；搜索区为输入框占位「请输入产品名称或编号」+ 蓝色「搜索」按钮；表格两列「产品名称」「产品编号」；底部分页：共 N 条、每页显示（下拉 10/20 等）、首页/上一页/第 x 页/下一页/尾页、当前页信息（如 1/5）；底部「取消」「确认」按钮（确认为主按钮样式）。
- **实施**：新增产品选择弹窗 FXML + Controller；弹窗内请求产品列表（分页、关键词映射到 query.name 或 query.pronumber）；表格选中后回传 pronumber→产品编号、name→产品名称给主界面；主界面 ComboBox 显示产品名称，Label 显示产品编号；开始采集后 `productComboBox.setDisable(true)` 已存在，保持即可。

### 3.2 生产单号

- 格式 YYYYMMDD+序号，可编辑，默认当天+001；开始采集后不可编辑（已实现）。可选：后端校验唯一性或仅前端校验格式。

## 四、开始采集门禁与当前任务写入

### 4.1 门禁顺序

1. **产品与单号**：已选产品、生产单号非空（前端校验）。
2. **码包**：后端提供门禁接口，如 `GET /api/shiwan/code-package/check`，返回小标/中标/大标是否均已导入（可查 CodePackageImport 或热表是否有数据）；任一类未导入则提示“请先导入对应码包”并禁止开始。
3. **设备**：检测盒码相机、箱码相机、剔除装置等是否在线（复用系统设置中的设备配置与现有设备检测逻辑，如 DeviceConnectionManager）；未通过则提示具体设备并禁止开始。
4. **二次确认**：上述通过后弹出确认框，展示产品、生产单号、包装比例「1垛 M 箱、1箱 N 盒」；用户确认后再执行“写当前任务 + 开始采集”。

### 4.2 写当前任务与开始采集

- 门禁通过后，前端调用后端接口，例如 `POST /api/shiwan/current-task/start`，请求体：orderNo、productNo、productName、boxesPerPallet（M）、boxesPerCase（N）。后端将当前订单号、产品编码等写入 ProductionOrder（当前任务），并返回当前任务标识（如 taskId）供后续盒箱关联、当前垛信息等使用。
- 前端收到成功后再：`isRunning=true`、按钮改为“停止采集”、规格与产品禁用、`runningStatusLabel` 显示“采集中”；并可选拉取一次「当前垛信息」以初始化当前箱数/当前盒数（见下）。

## 五、盒箱垛采集流程与数据源

### 5.1 盒箱关联（环节 4）

- **数据来源**：2 号机盒码相机 + 箱码相机上报盒码、箱码（设备回调或串口服务上报到前端/后端）；每 N 盒（如 4 盒）组成一箱时采集箱码。
- **后端**：提供盒箱关联接口，例如 `POST /api/shiwan/box-case/associate`，入参：当前任务标识、本箱 N 个盒码列表、箱码；后端逻辑：校验 N 盒与箱码一一对应、箱码在箱外码大标码包内、非重码 → 通过则建立盒-箱关联、箱码计数+1，返回成功及当前箱数/当前盒数；不通过则返回失败与原因。**说明**：先查码关系表该箱内是否有 Status=3（待剔除）的盒并执行剔除、写剔除记录表的逻辑**本期先不实现**。
- **前端**：设备上报的盒码/箱码汇总到“满 N 盒+1 箱码”时调用上述接口；根据返回在数据接收区追加「HH:mm:ss 盒箱关联 箱码xxx 成功」或「盒箱关联 箱码xxx 失败，原因」；更新当前箱数、当前盒数（以后端返回为准）；若后端返回“满箱”，前端追加「HH:mm:ss 满箱」；若返回“满垛”，走成垛与上传逻辑（见下）。

### 5.2 当前箱数、当前盒数、每垛箱数、每箱盒数

- **每垛箱数 / 每箱盒数**：绑定左侧采集规格输入框（开始采集后只读）；与 2.3 持久化一致。
- **当前箱数**：选任务/恢复时由「当前垛信息」接口返回；采集中由盒箱关联接口返回 currentCount；成垛/强制满垛后置 0。技术分析建议后端按「当前任务+当前垛」返回，前端展示；若后端统一返回，前端以接口为准，否则可在前端内存 +1/置 0 与接口做同步。
- **当前盒数**：每完成一次盒箱关联当前箱的盒数 +1，满 N 盒即满一箱时清零。技术分析建议**后端在盒箱关联接口中返回「当前箱已关联盒数」和「每箱盒数 N」**，前端直接展示，避免前后端不一致。

### 5.3 满垛与成垛（环节 5）

- 当前箱数达到 M 时，由后端在盒箱关联或单独成垛接口中生成**虚拟垛标**（格式见系统设置→业务→虚拟垛标规则），建立 M 箱与垛码关联。
- 前端：在收到成垛成功响应后，在数据接收区追加「HH:mm:ss 满垛提示」或「垛码 xxx 成垛」；已生产垛数 +1（或再调已生产垛数接口）；当前箱数/当前盒数置 0；实时上传数据区追加一条「垛 xxx、M箱、待上传」；若开启自动上传则触发上传（见下）。

### 5.4 数据接收区内容汇总

- 盒箱关联成功/失败（时间+盒码/箱码+结果/原因）；
- 满箱、满垛提示；
- 垛码 + 上传状态（待上传/上传成功/上传失败，原因）；
- **1 号机数据**：1 号机瓶盒数据通过 [2 号机系统设置](.cursor/plans/2号机系统设置补充实现_1d319ad3.plan.md) 中「连接」Tab 下的 **1 号机连接** 配置 + 后端定时（每 1 秒）将 T_Code（Status=0）同步写入 CodeRelationUpload（OrderNo/ProductNO 取当前 2 号机生产订单号与产品编码，AddTime 取当前时间）。主界面数据接收区若需区分来源，可 optional 展示「1号机数据已同步」等提示；否则与 2 号机本机采集数据一并展示。
- 关联失败/异常（重码、错码、盒箱不符等）：由接口返回 message/details，按 P02-01 报警描述规则写清码值与原因，并写入报警信息区（见下）。

## 六、实时上传数据区与自动上传

### 6.1 实时上传数据区数据源

- 需求：列表展示垛码、箱数、上传状态（待上传/已上传等），与 P02-08 一致或共享数据源。
- 实施：数据来源为 **CodeRelationUpload** 中“成垛未上传/已上传”的垛维度数据；该表数据除 2 号机本机盒箱垛采集写入外，还包括 **1 号机 T_Code 同步**写入的记录（见 2 号机系统设置计划：同步时 OrderNo/ProductNO 为当前 2 号机生产订单号与产品编码）。可由后端提供接口，如 `GET /api/shiwan/upload-list?orderNo=xxx` 或按当前任务、时间范围返回垛码、箱数、上传状态。主界面初始化或开始采集后、成垛后、上传结果回调后刷新该列表；与 P02-08 可共用同一接口或同一视图。

### 6.2 成垛后自动上传（环节 6）

- 配置：系统设置→业务→上传配置中「自动上传开关」为开时，成垛后自动上传该垛至米多。
- 接口：技术分析 3.3 明确 path 为 **api/sign/md.fc.Store/v1/SyncCodeAndVirtualRelation** 与 **GetSyncCodeAndVirtualRelationResult**；**完整 URL = base URL + path**，**base URL 与签名**从 2 号机系统设置（配置文件）读取，见 [2 号机系统设置 4.5](.cursor/plans/2号机系统设置补充实现_1d319ad3.plan.md)。因异步执行，需轮询结果接口查询上传状态。
- 实施：后端在成垛完成后，若自动上传开关为开，则使用配置中的 base URL 与签名调用上述开放平台上传接口并轮询结果；将结果（成功/失败及原因）回写库或通过 WebSocket/轮询接口给前端；前端更新实时上传数据区该垛状态，并在数据接收区追加「垛码 xxx 待上传/上传成功/上传失败，原因」；上传失败时触发报警（见下）。

## 七、报警与指示灯

### 7.1 报警规则（2 号机）

- 连续重码 3 次：**长报警**，需人工「关闭报警」；
- 上传异常（数据上传失败）：**长报警**，需人工「关闭报警」；
- 其他（如单次关联失败）：可为短报警或仅写报警信息区。
- 报警描述：按 P02-01 规则写具体原因（小标/中标/大标不通过、重码、错码、盒箱不符等），含码值与原因。

### 7.2 指示灯与蜂鸣

- 上传成功：绿灯常亮，约 1 分钟后自动熄灭；
- 上传失败：红灯常亮 + 蜂鸣，需人工点击「关闭报警」后停止；
- 关闭报警：向报警设备发送停止指令（协议按系统设置中设备配置），并重置界面报警状态（不亮、不响）。

### 7.3 前端

- 报警信息区：接口返回的失败原因写入 `alarmLogList`，样式按 LogType.ERROR（红）；连续重码 3 次与上传失败时触发长报警（调用设备控制或后端“触发报警”接口）。
- 关闭报警按钮：点击后调用后端或设备服务“关闭报警”，并刷新状态。

## 八、单位实时统计与剔除

### 8.1 已生产垛数

- 选任务/进入页时：调用后端按订单+产品统计，如 `GET /api/code/produced-pallet-count-by-product-optimized` 或石湾等价接口（OrderNo、ProductNO）；
- 每次成垛成功：前端本地 +1 或短时防抖后重新请求接口，保证与剔除、多端一致。P02-01 要求垛数可点击进入 P02-07，可在本阶段仅做数字展示与点击跳转 Tab。

### 8.2 总剔除数

- 数据源：剔除记录表，按当前生产单号（或任务）统计条数（或按箱码去重）。
- 后端：提供 `GET /api/stats/reject-count?orderNo=xxx`（或按 taskId）；剔除发生时由盒箱关联/剔除逻辑写 RejectRecord，该接口只读统计。
- 前端：定时（如每 2 秒）或事件驱动（每次剔除/报警后）调用并更新 `rejectCountLabel`；总剔除数 40px、红色加粗（已有样式）；点击可跳转 P02-07 剔除记录弹窗（若已实现）。

### 8.3 剔除数清零

- 仅界面将「总剔除数」显示清零，不删剔除记录表数据；后端无需接口，前端将 `totalRejectCount` 与 `rejectCountLabel` 置 0，并写操作日志。

## 九、任务控制与其它交互

### 9.1 强制满垛

- 未满 M 箱时也可结束当前垛，生成虚拟垛标并重置箱数计数。后端提供强制满垛接口，入参：当前任务标识、当前箱数；后端生成虚拟垛标、写关联、重置当前垛计数，返回垛码；前端更新已生产垛数、当前箱数/盒数置 0、实时上传区追加一条、数据接收区追加满垛提示，并若自动上传为开则触发上传。

### 9.2 提取工单未成垛

- 流程：弹窗输入/扫箱码 → 后端按箱码查订单（CodeRelationUpload 或任务表）→ 返回订单号、产品、未成垛箱数等 → 确认后回显主页面（产品、生产单号、当前箱数等）并允许继续生产。
- 后端：提供按箱码查任务/订单接口；可选“恢复当前任务”接口，将主界面状态恢复为该任务进行中、当前箱数等。
- 前端：新建提取工单未成垛弹窗，输入框支持手动与扫码；调用查询接口展示结果，确认后关闭弹窗并刷新主界面绑定数据。

### 9.3 收回剔除

- 向剔除装置发送收回指令（PLC），设备配置在系统设置；后端或前端通过现有设备控制服务发送，仅需在「收回剔除」按钮中调用并写操作日志。

### 9.4 无需采集码

- 开关打开时，读码设备不工作、不采集任何数据；前端可置“无码模式”标志，设备或后端收到该标志时不执行采集逻辑；界面与现有占位一致即可。

### 9.5 退出软件与启动恢复（暂存逻辑）

- 若存在未满垛数据（currentCases > 0），弹窗「存在未满垛(X箱)，是否暂存？」按钮：取消、强制满垛、暂存。
- **暂存**：不单独提供“暂存未成垛”接口；**将当前任务下未成垛对应的 CodeRelationUpload 相关数据的 Status 标记为 0（未完成）**（或使用约定字段标记未成垛），以便下次启动时识别。用户点击「暂存」后允许退出。
- **启动恢复**：每次打开 2 号机时，后端或前端**读取 CodeRelationUpload 中 Status=0 的未完成数据**（按当前产线/任务维度）；若存在则弹窗提示：「检测到上次退出软件时有未满垛数据，默认继续使用该垛数据继续采集」。该弹窗**仅有「确认」按钮和右上角关闭键**，仅起提示作用；用户点击确认或关闭后关闭弹窗，并**初始化该垛数据到采集界面**（如恢复当前箱数、当前垛标识、产品与生产单号等），继续采集。
- **强制满垛**：调用强制满垛接口，再允许退出。
- **取消**：关闭弹窗，不退出。
