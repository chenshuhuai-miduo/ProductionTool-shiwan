## 一、数据库设计（MySQL）

表结构以 [docs/石湾开发文档/数据库设计.md](docs/石湾开发文档/数据库设计.md) 为准，字段命名均为 **PascalCase**。已存在：码包导入记录表 `CodePackageImport`、码包码热表 `CodePackageItemHot`、码包码冷表 `CodePackageItemCold`（热表/冷表**无** Status、IsDeleted 等状态字段）。以下仅补充计划中依赖的**拉取时间表**与**全码视图**，并约定与现有表字段的对应关系。

### 1.1 码包导入记录表（已存在，仅引用）

表名：`CodePackageImport`。主要字段：`Id`, `PackageType`, `PackageName`, `ImportTime`, `ImportSource`, `CodeCount`, `Status`, `Remark`, `FileName`, `CreateTime`, `UpdateTime`。Status：1 正常 / -1 删除。索引见数据库设计文档。

### 1.2 码包码热表（已存在，仅引用）

表名：`CodePackageItemHot`。字段：`Id`, `ImportId`, `PackageType`, `CodeValue`, `CreateTime`。无 Status/IsDeleted。唯一约束 `UkPackageCode`(PackageType, CodeValue)。

### 1.3 码包码冷表（已存在，仅引用）

表名：`CodePackageItemCold`。字段：`Id`, `ImportId`, `PackageType`, `CodeValue`, `AssociatedAt`, `CreateTime`, `UpdateTime`。无 Status/IsDeleted。

### 1.4 上次成功拉取时间表 `CodePackagePullTime`（需补充到数据库设计文档）

按关系类型记录开放平台上次成功拉取时间，用于增量请求。仅 `relationship_type` 1（小标）、3（大标）有记录。

```sql
CREATE TABLE `CodePackagePullTime` (
    `Id` BIGINT NOT NULL AUTO_INCREMENT COMMENT '主键',
    `PackageType` TINYINT NOT NULL COMMENT '码包类型：1 小标 / 3 大标（与开放平台 relationship_type 一致）',
    `LastPullTime` DATETIME NOT NULL COMMENT '该类型上次成功拉取时间',
    `UpdateTime` DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    PRIMARY KEY (`Id`),
    UNIQUE KEY `UkPackageType` (`PackageType`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='码包在线拉取上次成功时间（按类型）';
```

### 1.5 全码视图（用于快速判断重码，需补充到数据库设计文档）

热表、冷表均无 IsDeleted/Status，视图为两表并集，字段 PascalCase：

```sql
CREATE VIEW `AllCodePackageCodes` AS
SELECT `PackageType`, `CodeValue` FROM `CodePackageItemHot`
UNION ALL
SELECT `PackageType`, `CodeValue` FROM `CodePackageItemCold`;
```

导入时用该视图 + 批量 IN 查询判断本批码是否已存在（见第二节）。重码查询示例：`SELECT CodeValue FROM AllCodePackageCodes WHERE PackageType = ? AND CodeValue IN (?, ?, ...)`。

### 1.6 关联时间数据来源

关联状态/关联时间：该码是否已被采集关联 + 首次关联时间。来自 **CodeRelationUpload** 的 `AddTime`；按码包类型对应字段：小标→SmallSerialNumber，中标→MediumSerialNumber，大标→BiggerSerialNumber/BigSerialNumber。石湾环境表已在 MySQL，可直接同库按码查该码最小 AddTime。

## 二、导入时快速判断重码方案

需求：按**文件名 + 内容**判断，避免重复或高度重叠入库；需查热表+冷表（或统一视图）。

**推荐做法：all_codes 视图 + 分批 IN 查询**

1. **文件名**：同一 `PackageType` 下若存在未删除的导入记录（`CodePackageImport.Status = 1`）且 `FileName` 相同，可先判为“同文件”重复，直接拒绝或提示。
2. **内容**：将本次解析得到的码集合**去重**后，按批（如每批 500～1000 条）执行：
  - `SELECT CodeValue FROM AllCodePackageCodes WHERE PackageType = ? AND CodeValue IN (?, ?, ...)`
  - 将返回的 `CodeValue` 与本批求交集，得到“已存在”的码。
3. **去重与重叠率**：对整次导入的所有码做去重后，统计“已存在码数 / 本批去重总数”。若重叠率超过阈值（如 80%）视为“高度重叠”，拒绝导入并提示；若存在部分重复但未达阈值，可提示“存在 N 条重码，是否仅导入新码？”由产品决定是仅导入新码还是整批拒绝。
4. **性能**：IN 列表长度控制在 500～1000，避免 SQL 过长；视图在 MySQL 上等价于两表 UNION，对已有索引（PackageType, CodeValue）友好。若单类型码量极大（十万级），可后续再考虑在应用层用 Bloom 过滤器做“可能不存在”的快速过滤，再对候选做一次 DB 校验。

计划实施时在 **Application 层** 封装“按文件名 + 本批码集合判断是否重复/高度重叠”的方法，Infrastructure 层提供“根据 PackageType + CodeValue 列表批量查已存在”的查询（基于 `AllCodePackageCodes` 或等价 SQL）。

## 三、在线更新流程（开放平台）

- **接口**：开放平台 **POST** `/api/v1/code/package/query-completed`。**不传 type**，请求体为时间范围。
- **请求参数**（JSON body）：
  - `start_time`（string，必传）：查询开始时间，格式 `yyyy-MM-dd HH:mm:ss`；归一化为当天 00:00:00。
  - `end_time`（string，可选）：查询结束时间；有值时归一化为当天 23:59:59，为空时服务端使用当前时间 +6 小时。
- **响应结构**：`return_code`（0 成功）、`return_msg`、`return_data`。`return_data` 含 `total_count`、`data`（数组）。`data[]` 单项：`id`（任务ID/文件MD5）、`file_name`、`relationship_type`（1 小标二维码 / 2 导购码营销码 / 3 大标动销码）、`success_count`、`failed_count`、`upload_time`、`end_process_time`、`file_download_address`（仅 relationship_type 为 1 或 3 且上云成功时有值）。
- **流程**：
  1. 按类型（仅小标、大标）读取**上次成功拉取时间**：从表 `CodePackagePullTime` 取 `PackageType=1` 与 `PackageType=3` 的 `LastPullTime`；若无记录则使用一个较早的默认时间（如系统上线日 00:00:00）作为 `start_time`。
  2. 调用开放平台接口（URL 配置项 `code.package.openplatform.url`），请求体：`{"start_time":"yyyy-MM-dd HH:mm:ss","end_time":"..."}`（可选）。不传 type。
  3. 解析响应：若 `return_code !== "0"` 则本次整体失败，不更新任何 `LastPullTime`；否则遍历 `return_data.data`，筛选 `relationship_type in (1, 3)` 且 `file_download_address` 非空的项。
  4. 对每一项：根据 `relationship_type` 映射为本地码包类型（1→盖外码小标，3→箱外码大标），使用 HTTP 客户端按 `file_download_address` **下载文件**到临时文件或内存流，按行解析 TXT 得到码列表。
  5. **重复判断**：按“文件名（若有）+ 内容”查 `AllCodePackageCodes` 视图，高度重叠则跳过该项并记录失败原因，**不更新**该类型 `CodePackagePullTime.LastPullTime`。
  6. 不重复则：写入 `CodePackageImport`（ImportSource=在线）、批量写入 `CodePackageItemHot`，并更新该类型在 `CodePackagePullTime` 的 `LastPullTime`（取该任务的 `upload_time` 或 `end_process_time`）。
  7. 某类型失败（网络、解析、重复等）：**不更新**该类型 `LastPullTime`，**不覆盖**本地已有数据；界面提示该类型失败原因。
- **启动时**：2 号机启动后自动执行一次在线更新，前端在主窗加载完成后调一次 `POST /api/code-package/import/online`（不阻塞界面）；失败时仅提示或打日志，不更新各类型时间戳（由后端保证）。

## 四、本地导入与删除

- **本地导入**：三种类型均可；盒外码中标仅能本地导入（前端下拉限制）。步骤：选择码包类型 → 选择 TXT 文件 → 输入密码（默认 123456，可配置）→ 后端校验密码 → 解析 TXT → 文件名+内容重复判断（基于 `CodePackageImport.FileName` + `AllCodePackageCodes`）→ 写 `CodePackageImport` + `CodePackageItemHot`。
- **删除**：仅当该码包（该 `ImportId`）内**无任何已关联码**时可删。校验方式：该 `ImportId` 下是否存在任一条码在 `CodePackageItemCold` 或 CodeRelationUpload 中出现；存在则禁止删除并提示。删除为**物理删除**：`DELETE` 该条 `CodePackageImport` 记录，并 `DELETE` 该 `ImportId` 在 `CodePackageItemHot` 中的全部记录；**不删除** `CodePackageItemCold` 中该导入的冷数据（保留审计）。热表、冷表无 Status 字段，不做状态更新。
- **使用完**：码包内码全部落冷后，导入记录仍保留，**不可删除**（供码替换等使用），删除校验时按“存在已关联码则不可删”统一处理。

## 五、后端实现要点

### 5.1 模块与分层

- **miduo-entity**：枚举（码包类型、导入方式、状态）、DTO（分页查询请求、导入记录 VO、查看码 VO、本地导入请求等）。
- **miduo-infrastructure**：  
  - MySQL：PO 与 Mapper 对应表名/字段 PascalCase（`CodePackageImport`, `CodePackageItemHot`, `CodePackageItemCold`, `CodePackagePullTime`）；基于视图 `AllCodePackageCodes` 的“批量查已存在码”方法。  
  - 开放平台：HTTP 客户端（POST query-completed，请求体 start_time/end_time；解析 return_data.data，根据 file_download_address 下载文件），URL 由配置提供。
- **miduo-application**：`CodePackageApplicationService`：分页列表、在线更新（调开放平台 start_time/end_time → 解析 data[] → 按 file_download_address 下载→解析→重码判断→入库→更新 CodePackagePullTime）、本地导入（密码校验、解析、重码判断、入库）、删除（校验后**物理删除** CodePackageImport 及该 ImportId 在 CodePackageItemHot 中的记录）、查看某码包下码列表（分页+搜索，带关联状态与关联时间）。
- **miduo-controller**：`CodePackageController`，REST 接口（见下）。若项目有石湾 variant 且仅 2 号机需要，可挂在该 variant 的 Controller 包下，或统一放在主 controller 包由 2 号机启动器加载。

### 5.2 配置项

- 开放平台 base URL：`code.package.openplatform.url`（写死配置文件）。
- 本地导入密码：`code.package.import.password`（可配置，默认 123456）。
- 可选：下载超时、重码重叠率阈值等。

### 5.3 REST 接口约定

- `POST /api/code-package/page`：分页列表，请求体含 keyword、导入时间起止、导入方式、码包类型、状态、pageNum、pageSize；返回 `PageOutput<CodePackageImportVO>`。
- `POST /api/code-package/import/online`：触发在线更新（仅盖外码小标、箱外码大标）；返回汇总结果（成功/失败类型及原因）。
- `POST /api/code-package/import/local`：本地导入，multipart 或 JSON+文件内容；参数：packageType、fileName、password、codes（或 file）；返回导入结果。
- `GET /api/code-package/{id}/codes`：查看某码包下码列表，支持 keyword、pageNum、pageSize；返回 `PageOutput<CodePackageViewCodeVO>`（码值、关联状态、关联时间）。
- `DELETE /api/code-package/{id}`：**物理删除**，校验“该包内无已关联码”；否则 4xx + 提示信息。删除时删除该条导入记录及该 ImportId 在热表中的全部行，冷表不删。

### 5.4 落冷时机

采集/关联成功时（写入 CodeRelationUpload 时），在应用层或领域层：根据码类型（小标/中标/大标）在对应热表 `CodePackageItemHot` 中查找该码，若存在则写入冷表 `CodePackageItemCold`（AssociatedAt 取当前关联记录的 AddTime），并将热表该条**物理删除**（热表无 Status 字段）。落冷逻辑与现有采集流程在同一事务或同一调用链中完成。

## 八、与采集/码替换的衔接（简要）

- **采集**：瓶/盒/箱码校验时查**热表** `CodePackageItemHot` 判断“在码包内”；通过后写关联表，并触发**落冷**（热表该条物理删除，冷表写入 AssociatedAt）。
- **码替换**：新码须在对应类型码包内且未使用，同样查热表；替换成功后若该码原在热表则落冷。
