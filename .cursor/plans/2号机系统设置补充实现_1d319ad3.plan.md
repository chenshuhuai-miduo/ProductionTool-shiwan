## 三、虚拟垛标序号持久化（仅 2 号机）

- 需求：序号须持久化，**从数据库查询当天同产线号最大序号+1**，避免重启后从 001 重复导致重码。  
- **方案评估**：  
  - **方案 A（查 CodeRelationUpload）**：按 `VirtualSerialNumber` 解析后按日期+产线号取 max(序号)+1。实现简单、无需新表，但当 **CodeRelationUpload 数据量很大**时，每次成垛都扫/解析该表会带来性能与索引压力，且 VirtualSerialNumber 格式若后续调整会影响解析。  
  - **方案 B（独立小表）**：专门存“日期+产线号+当前序号”，每次成垛时按主键查询并更新（或 CAS 递增）。**数据量大时更合适**：访问量小、表小、语义清晰、不依赖大表结构与解析，易维护。
- **结论**：采用 **方案 B，新增小表**。表名使用 **PascalCase**：`VirtualPalletSequence`。字段建议：日期（如 `SeqDate` DATE）、产线号（如 `LineCode` VARCHAR）、当前序号（如 `CurrentSeq` INT），主键（`SeqDate`, `LineCode`），每次成垛时查询并更新该表取下一序号。  
- **系统设置界面**：只负责保存/展示**前缀、产线号**；**序号、时间、随机数**为只读说明，不在此界面编辑。成垛生成虚拟垛标时，由**成垛逻辑**（主流程或后端）读取配置的前缀/产线号，并查 `VirtualPalletSequence` 取下一序号，生成 VirtualSerialNumber。  
- 若当前成垛/上传流程尚未接好 2 号机库，需在“成垛时生成虚拟垛标”的流程中接入：读取系统设置中的前缀与产线号 + 上述小表序号逻辑。

## 四、各功能点实现要点

### 4.1 业务 Tab

- **码位数配置**：保存到配置文件；采集/校验逻辑（主界面或后端）读取该配置做位数校验（-1 不校验）。  
- **虚拟垛标规则**：仅持久化**前缀、产线号**；界面时间/序号/随机数为只读说明；成垛时用“前缀+YYYYMMDD+序号(3位)+产线号+2位随机”生成，序号来源见第三节。  
- **上传配置**：自动上传开关写入配置；成垛后是否自动上传由该开关控制（主流程或上传模块读取配置）。

### 4.2 页面配置 Tab

- 8 行 Switch 与配置项一一对应（顺序固定为文档所列）；“数据采集（盒箱垛）”行禁用 Switch。  
- 保存时：将 8 项的开关状态及**顺序**（如按勾选顺序存为列表）写入配置文件。  
- **主界面** [ShiwanM2MainController](miduo-frontend/src/main/java/com/miduo/cloud/frontend/controller/ShiwanM2MainController.java)：启动时读取页面配置，按**顺序**与**开关**动态显示/隐藏 Tab，且“数据采集（盒箱垛）”始终显示且不可关。

### 4.3 设备 Tab

- 需求注明“无需重复实现”设备表单与连接测试逻辑；可与现有 [SystemConfigController](miduo-frontend/src/main/java/com/miduo/cloud/frontend/controller/SystemConfigController.java) / DeviceConnectionManager 等对接。  
- 本阶段可做：**打印机、报警**相关字段写入 2 号机配置文件，保存/加载与界面绑定；**IO 设备**若已有独立配置（如 INI），可只做“从 2 号机配置中读取设备列表路径或启用开关”，具体设备增删改仍走现有模块。

### 4.4 连接 Tab

- **2 号机数据库连接**：2 号机确认读**本地 MySQL**；表单项（地址、端口、库名、用户名、密码）持久化到 2 号机 JSON 配置文件。**连接测试**通过**后端接口**完成（前端调用后端提供的“测试数据库连接”接口，后端用当前配置或传入参数做 JDBC 连接尝试，返回成功/失败与原因）。保存后：后端连接 DB 时读取该配置（或由后端在启动时加载同一 JSON 配置文件）。  
- **1 号机数据库连接（新增）**：在连接 Tab 下将 **「1 号机连接」做成独立子 Tab**（与「数据库连接」并列），表单项：**数据库地址**、**端口**、**数据库名**、**表名**（默认 T_Code）、**用户名**、**密码**。持久化到 2 号机 JSON 配置文件；用于下文「1 号机 T_Code 同步」任务通过 JDBC 连接 1 号机 SQL Server。可选：提供「测试 1 号机连接」按钮，调用后端用当前配置做 SQL Server JDBC 连接测试并返回结果。

### 4.5 接口配置（技术分析 268-282）

- 在配置文件中增加**开放平台 base URL**（IP/域名，如 `https://openapi.weixin12315.com`）及**签名相关配置**（如 appKey、appSecret 或签名算法参数，供调用带签名的 API 使用；具体字段以开放平台文档为准）。接口 path（如 `/api/sign/md.shop.products/v1/list`、SyncCodeAndVirtualRelation、GetSyncCodeAndVirtualRelationResult、码包 query-completed 等）可写死或一并配置。

## 五、1 号机 T_Code 数据同步（新增）

- **目的**：2 号机每隔 **1 秒**通过 JDBC 连接 1 号机的 SQL Server，读取指定表 **T_Code** 中未同步的数据，写入 2 号机 **CodeRelationUpload**，使 1 号机采集的瓶盒数据在 2 号机侧可见并可参与上传等逻辑。
- **逻辑约定**（与 [P02-01 主界面计划](.cursor/plans/p02-01_主界面盒箱垛关联补充实现_88970c27.plan.md) 一致）：① **2 号机启动后**先检测连接 1 号机 SQL Server 是否成功；② **若连接不成功，则不允许 2 号机进行采集**（门禁或采集使能前需通过 1 号机连接检测）；③ **检测连接成功后**，再按每 1 秒轮询 1 号机并同步写入 CodeRelationUpload。
- **1 号机表结构**（SQL Server）：  
`T_Code`：SerialNo（主键，自增）、BagCode、BagTime、BoxCode、BoxTime、Ratio、Ratio2、Sort、MyProductId、ProductionOrder、BatchId、LineId、Operator、Status、Status2、Other1、Other2、IsBiggest、MidCode 等；默认 Status=0、Status2=0、IsBiggest=0。
- **同步逻辑**：  
  1. **增量条件**：每次查询 `SerialNo > 上次同步的最大 SerialNo` 且 **Status=0** 的记录。
  2. **持久化进度**：将本次同步到的最大 **SerialNo** 写入配置文件或小型表（如 key=lastSyncedM1SerialNo），下次同步时用该值作为下限。
  3. **映射与写入**：对每条符合条件的 T_Code 记录，插入 2 号机 **CodeRelationUpload**：
    - **SmallSerialNumber** ← T_Code.BagCode  
    - **MediumSerialNumber** ← T_Code.BoxCode  
    - **OrderNo** ← **当前 2 号机的生产订单号**，**ProductNO** ← **2 号机选择的产品编码**，**AddTime** ← **当前时间**；**Status**=0（未完成）；**Type**=1（有箱码）；其余必填字段（BiggerSerialNumber、BigSerialNumber、VirtualSerialNumber、TagNo、DxCode、SalesCode 等）按 **2 号机采集规则** 赋默认值或从当前任务/配置获取。
  4. **执行频率**：定时任务每 **1 秒**执行一次（调度可放在后端；若 2 号机为纯前端则需与后端约定由后端提供“同步 1 号机 T_Code”接口并由前端或后端定时调用）。
- **配置来源**：1 号机连接信息（地址、端口、数据库名、表名、用户名、密码）从 2 号机系统设置「连接」Tab 下的 **1 号机连接配置** 读取；未配置或连接失败时本轮跳过并写日志，不阻塞 2 号机主流程。
- **依赖**：连接 Tab 下已新增 1 号机连接配置（见 4.4）；需后端提供 JDBC 连接 1 号机 SQL Server、执行上述查询与 CodeRelationUpload 插入的能力（或封装为单接口供定时任务调用）。

## 六、实现顺序建议

1. **配置层**：定义 2 号机配置 DTO/Bean 与配置文件路径；实现 **JSON** 读写，保证启动时加载、保存时覆盖；配置项包含 2 号机 DB、**1 号机 DB（地址、端口、库名、表名、用户名、密码）**、**上次同步最大 SerialNo**、**开放平台 base URL 与签名**。
2. **Controller 与界面**：在 `ShiwanM2SystemSettingsController` 中，初始化时从配置加载并绑定到各控件；**连接 Tab** 下将 **「1 号机连接」做成独立子 Tab**，表单项（地址、端口、库名、表名、用户名、密码）及保存/测试逻辑；各“保存”按钮写回配置并刷新内存中的配置（若有全局 ConfigHolder）。
3. **数据库连接测试**：前端调用**后端接口**进行 2 号机 MySQL 连接测试，结果展示在 `dbTestResultLabel`；可选增加「测试 1 号机连接」调用后端对 1 号机 SQL Server 的 JDBC 测试。
4. **主界面 Tab**：`ShiwanM2MainController` 启动时读取页面配置，按顺序与开关构建/显示 Tab；若已有 Tab 列表，则根据配置过滤并排序。
5. **虚拟垛标序号**：新增表 **VirtualPalletSequence**（PascalCase），在成垛流程中按日期+产线号查询并更新该表取下一序号，并读取系统设置中的前缀、产线号生成 VirtualSerialNumber。
6. **接口配置加载**：启动时把**开放平台 base URL 与签名**加载到内存，供产品列表（/api/sign/md.shop.products/v1/list）、码包、上传等模块使用；**IP 与签名均从配置文件读取**。
7. **1 号机 T_Code 同步**：后端实现定时任务（每 1 秒）或提供「同步 1 号机 T_Code」接口供调度：用 1 号机连接配置 JDBC 查询 T_Code（SerialNo > lastSyncedSerialNo 且 Status=0），将 BagCode→SmallSerialNumber、BoxCode→MediumSerialNumber；OrderNo/ProductNO 取**当前 2 号机生产订单号/选择的产品编码**，AddTime 取**当前时间**；Status=0、Type=1，其余按 2 号机采集规则写入 CodeRelationUpload，并更新 lastSyncedSerialNo。
8. **密码**：可选将系统设置密码改为从配置读取，默认 123456。

## 八、已确认结论

- **配置文件格式**：采用 **JSON**。  
- **数据库**：2 号机读**本地 MySQL**；**连接测试**通过**后端接口**实现。**1 号机**：在连接 Tab 下将「1 号机连接」做成**独立子 Tab**，配置 1 号机 SQL Server（地址、端口、库名、表名、用户名、密码），用于 T_Code 每 1 秒增量同步至 CodeRelationUpload（SerialNo > lastSyncedSerialNo 且 Status=0；BagCode→SmallSerialNumber，BoxCode→MediumSerialNumber；OrderNo/ProductNO/AddTime 取当前 2 号机生产订单号、选择的产品编码、当前时间；Status=0、Type=1，其余按 2 号机采集规则）。  
- **虚拟垛标序号**：采用**独立小表 VirtualPalletSequence**（PascalCase），避免 CodeRelationUpload 数据量大时扫表解析带来的性能与维护成本。  
- **开放平台接口**：base URL（IP）与签名写入配置文件，启动时加载，供 `/api/sign/md.shop.products/v1/list` 等接口调用。

可按上述顺序在 2 号机分支内实现；若有现有“成垛/虚拟垛标”代码位置，可把序号持久化与配置读取写进同一处。
