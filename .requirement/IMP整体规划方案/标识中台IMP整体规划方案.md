# 标识中台（IMP）整体规划方案

## 文档说明

标识中台（IMP）是一个长期建设工程，需要拆分多个阶段分期推进建设。标识中台（IMP）总体规划属于纲领性的指导方案，基于分级提灌原理，明确项目目标、定义产品架构及流程，制定实施策略及路线图等，指导后续每期方案的建设，起到提纲挈领、承上启下的作用，确保“把事情做正确”；此外，总体规划蓝图会在后期的建设过程中持续迭代完善内容。

## 修订说明（2026-05-30）

本版基于《IMP 的部署方式及单/多租户架构设计方案 V1》和《标识中台 IMP 整体规划方案 V1.0 业务专家评审意见 V1》进行修订，重点补充：

- 三类标准部署方式
- 可扩展部署 Profile
- 平台层差异化能力
- 厂商级多租户与客户内部多租户模型
- 混合云扩展预留
- 一物一码领域的码生命周期、码关系、包材协同、扫码解析、业务路由和风控模型

整体规划方案定位为长期产品蓝图，不承载具体版本交付范围；具体版本范围、里程碑和项目突进计划另行在专项计划中定义。

## 第1章 产品定位与目标

本章聚焦产品定义与核心目标。关于目标市场、客户画像、竞争分析等内容，详见市场需求文档（MRD）。

### 1.1 产品定义

标识中台（Identity Middle Platform, IMP）是面向预包装品牌商的一物一码标识数字化基础设施，以“码”为核心数据载体，为企业建立统一的标识事实源和标识能力底座，通过标准 API/SDK 向多套第三方业务系统提供发码、存码、查码、管码等标识服务能力，支撑企业在防伪溯源、渠道管控、营销互动、生产协同等业务场景中的标识生成、标识治理、扫码解析和业务路由需求。

一家企业可能存在多子公司、多品牌、多事业部、多 SKU，也可能同时采购多套防窜物流、智能营销、互动营销、追溯、防伪等业务系统。IMP 作为标识中台，原则上应在企业内统一建设和统一订购，不与任一具体业务系统强绑定；BDE 只是 IMP 可对接的第三方业务系统之一，不能与 IMP 形成产品内置耦合。

#### 交付与部署形态

- **一套产品底座**：核心业务能力统一建设，包括发码、存码、查码、管码、主数据、权限、审计、接口治理等能力。
- **三类标准部署方式**：SaaS 公有云、专属云/私有云、OP 本地化。
- **可扩展部署 Profile**：前期规划 SaaS Profile、Dedicated/Private Profile、OP Profile；后期如需支持混合云，可扩展 Hybrid Profile。
- **两类多租户口径**：SaaS 公有云采用厂商级多租户；专属云/私有云和 OP 本地化支持客户内部多租户。
- **多业务系统对接**：同一企业的一套 IMP 可同时对接多套第三方业务系统，通过统一 API 网关、系统凭据、业务路由和事件回传机制完成解耦集成。
- **规划边界**：本方案定义长期产品架构、部署口径和能力路线；具体版本交付范围以对应项目计划或迭代计划为准。

### 1.2 产品目标

1. **打造独立可交付的标识基础设施**：建设可独立运行的标识服务系统，形成可面向市场售卖的标准产品能力，并为 SaaS 公有云、专属云/私有云、OP 本地化及后续混合云扩展保留统一产品底座、租户模型、授权模型和平台层适配能力。
2. **建立可配置的差异化产品能力**：通过流程配置与多场景适配机制覆盖主流业务差异，降低定制开发比例与交付周期。
3. **建立统一的标识事实源**：统一管理码生命周期、码关系、码状态、码包、批次和扫码事件，确保防伪、溯源、营销、渠道和生产协同系统基于同一套标识数据工作。
4. **构建开放的生态协同体系**：开放标准 API/SDK 连接包材厂、工厂、设备商及多套第三方业务系统，实现品牌商与生态伙伴的高效协同。

### 2.2 核心业务场景概览

标识中台围绕标识全生命周期，划分为以下 7 个核心业务环节：

| 环节 | 场景定位 | 关键角色 |
| --- | --- | --- |
| 前置准备 | 标识生命周期起点，完成主数据/模版/规则配置，支持第三方码导入 | 品牌商IT、运营人员 |
| 申请码包 | 发码申请→审批→生码，完成码包生成与分发准备 | 品牌商运营、包材对接人员 |
| 包材赋码 | 包材厂下载码包→赋码→采集→回传，实现赋码数据可追溯 | 包材厂对接人员、生产人员 |
| 包材签收 | 品牌商确认包材/验收，采购人员收货确认 | 品牌商运营、采购人员 |
| 采集关联 | 产线采集各包装层级间的父子关联关系（瓶→盒→箱→垛），形成完整的包装单元结构 | 工厂生产人员 |
| 成品入库 | 标识与商品绑定、批次激活，使码可被业务系统调用 | 品牌商运营、仓库人员 |
| 扫码解析与业务路由 | 对扫码请求进行码解析、状态校验、风险判定和业务系统路由 | 消费者、稽查人员、第三方业务系统 |

### 2.3 场景流转关系

```text
前置准备 → 申请码包 → 包材赋码 → 包材签收 → 采集关联 → 成品入库 → 扫码解析与业务路由
                                              ↑
                                        （可选，层级码场景）
```

**说明**：采集关联环节仅在层级码场景下触发，单码/同级码场景可跳过直接进入成品入库。扫码解析与业务路由不是某个单一业务系统的功能，而是 IMP 面向防伪、追溯、营销、渠道等第三方业务系统提供的统一标识入口能力。

## 第3章 产品架构设计

### 3.1 产品架构

#### 3.1.1 产品架构图

> 图：标识中台（IMP）产品架构图

##### 架构分层说明

标识中台采用六层架构，按“用户→入口→业务→服务→平台”自上而下分层，治理层作为横切能力贯穿全局。

| 层级 | 是什么 | 包含内容 | 为什么独立 |
| --- | --- | --- | --- |
| 用户层 | 使用 IMP 的角色与外部系统 | 品牌运营人员、包材厂人员、采购人员、工厂人员、外部系统 | 明确服务对象，区分人机与系统集成 |
| 访问层 | 用户交互入口 | PC 端、小程序、APP、API/SDK | 统一接入管控，隔离前端形态差异 |
| 业务层 | 标识生命周期业务模块 | 发码模块、赋码模块、采集模块、成品模块 | 按业务流程组织，职责边界清晰 |
| 服务层 | 业务能力的技术支撑 | 标识中心（码生成与存储）、订单中心（发码订单管理）、批次中心（发码批次与成品批次管理）、配置中心（模版与规则）、主数据中心（基础数据）、解析路由中心（扫码解析与业务路由） | 能力复用，业务变化不影响底层服务 |
| 平台层 | 部署与运营基础能力 | SaaS 账户与计费、专属/私有实例管理、OP 系统管理与 License、统一租户模型、部署 Profile | 隔离部署形态差异，核心业务能力共享复用 |
| 治理层 | 横切能力 | 身份权限（访问控制）、审计追责（操作留痕）、安全管控（风控策略）、数据质量（校验规则）、异常闭环（问题处理）、接口治理（API 管控）、运营度量（指标监控） | 通用能力抽离，避免重复建设 |

##### 层级协作关系

- **调用链路**：用户层 → 访问层 → 业务层 → 服务层，平台层提供运营基础
- **治理横切**：治理层不在调用链上，而是贯穿各层提供权限、审计、安全等能力

#### 3.1.2 架构设计说明

架构设计要点：

1. **分层解耦**：业务层与服务层分离，业务流程变化不影响底层服务能力，降低系统迭代风险。
2. **模块划分**：业务层按“发码→赋码→采集→成品”四大业务模块划分，对应标识从生成到可用的完整生命周期，职责边界清晰。
3. **治理贯穿**：治理层不是独立业务层，而是横切能力集合（权限、审计、风控、数据质量等），确保全局一致性与合规性。
4. **部署兼容**：平台层通过可扩展部署 Profile 支撑不同部署形态。SaaS Profile、Dedicated/Private Profile 与 OP Profile 面向三类标准部署方式；后期如需支持混合云，可扩展 Hybrid Profile。核心业务能力共享复用，差异集中在账户计费、实例管理、License、系统运维、本地节点和数据同步策略。
5. **租户兼容**：租户模型同时支持厂商级多租户与客户内部多租户。SaaS 公有云面向多个外部品牌商租户；专属云/私有云和 OP 本地化面向单客户独立实例，并可在客户实例内划分品牌、事业部、子公司、工厂等内部租户。
6. **多系统解耦**：IMP 是企业级标识中台，第三方业务系统通过统一 API 网关、系统凭据、路由规则和事件机制接入；BDE、防窜系统、营销系统、追溯系统等均是外部能力消费方，不应与 IMP 内置耦合。
7. **能力承载**：第1章定义的“发、存、查、管”四类核心能力，由业务层四大模块和解析路由能力协同实现，并通过服务层各中心提供统一支撑。

### 3.2 部署方式与单/多租户架构

部署方式和租户架构是两个维度：

| 概念 | 回答的问题 | IMP 中的设计口径 |
| --- | --- | --- |
| 部署方式 | 系统部署在哪里、谁运维、数据在哪里、如何交付 | 标准口径为 SaaS 公有云、专属云/私有云、OP 本地化；后期可扩展混合云 |
| 租户架构 | 一套系统服务多少组织，以及数据、权限、配置如何隔离 | 单租户、厂商级多租户、客户内部多租户 |

#### 3.2.1 标准部署方式

| 对比维度 | SaaS 公有云 | 专属云/私有云 | OP 本地化 |
| --- | --- | --- | --- |
| 部署形态 | 部署在厂商统一云平台 | 为单客户部署独立云实例，可位于厂商专属云、客户指定云或客户私有云 | 部署在客户本地机房、内网或客户完全控制的服务器环境 |
| 实例形态 | 一套平台服务多个外部客户 | 一个客户一套独立实例 | 一个客户一套本地实例 |
| 典型租户模型 | 厂商级多租户 | 单客户实例 + 客户内部多租户 | 单客户实例 + 可选客户内部多租户 |
| 售卖方式 | 订阅、发码券、套餐 | 订阅/项目制/资源费/托管费 | License、实施费、运维服务费 |
| 运维责任 | 厂商统一运维 | 厂商托管或双方协作 | 客户 IT 为主，厂商交付支持 |
| 升级方式 | 厂商统一升级 | 独立实例按约定窗口升级 | 客户确认后本地升级 |
| 数据主权 | 平台侧托管，按租户逻辑隔离 | 客户级实例隔离，数据位于独立云环境 | 客户本地掌控，数据不出客户内网 |
| 目标客户 | 快速上线、轻资产运营、标准化需求客户 | 中大型客户、集团客户、数据隔离要求高的客户 | 数据主权、等保合规、自主可控要求强的大型客户 |

**规划边界说明**：上表是产品长期部署口径，不代表任一具体版本需要同时完整交付三种部署方式。具体版本的交付范围应在对应项目计划或迭代计划中单独定义。

#### 3.2.2 单租户与多租户关系

| 租户架构 | 含义 | 典型部署 |
| --- | --- | --- |
| 单租户 | 一套实例只服务一个客户组织，不在系统内继续划分租户 | OP 本地化、小型私有化项目 |
| 厂商级多租户 | 一套系统同时服务多个外部品牌商客户，各客户之间数据、权限、配置隔离 | SaaS 公有云 |
| 客户内部多租户 | 一套客户独立实例内，继续服务该客户集团内部多个组织、品牌、事业部、工厂 | 专属云/私有云、OP 本地化、大客户专属实例 |

需要避免将“SaaS = 多租户、OP = 单租户”简单绑定。更准确的口径是：SaaS 公有云通常是厂商级多租户；专属云/私有云和 OP 本地化通常是单客户独立部署，但系统内部仍可支持客户集团内的多租户。

#### 3.2.3 管理后台分层

IMP 的后台应按管理对象分层，而不是混成一个后台：

| 后台类型 | 管理对象 | 典型使用者 | 部署位置 |
| --- | --- | --- | --- |
| 厂商运营后台（米多星球） | 客户开户、合同、套餐、订购、续费、发票、实例、版本、授权 | 厂商运营、商务、客服、交付 | 厂商侧 |
| 厂商 License/授权中心 | 授权有效期、模块开关、码量额度、并发数、版本许可、License 文件生成 | 厂商授权管理员、商务、交付 | 厂商侧 |
| 系统管理控制台 | 客户实例本身的运行、授权、服务、审计、备份、升级 | 客户 IT、运维、厂商交付 | 客户实例内 |
| 客户实例管理后台 | 客户实例内的租户、组织、品牌、工厂、供应商、账号、角色、权限、额度分配 | 客户超级管理员、业务管理员 | 客户实例内 |

**判断标准**：

- 会影响整套实例运行、安全、授权、升级、备份的，放在系统管理控制台；
- 只影响某个内部租户、组织、品牌、工厂、账号、权限或业务额度的，放在客户实例管理后台。

OP 本地化场景下，两者的边界尤其重要：

| 对比项 | 系统管理控制台 | 客户实例管理后台 |
| --- | --- | --- |
| 管理对象 | OP 实例本身 | OP 实例内的业务使用 |
| 典型使用者 | 客户 IT、运维、厂商交付 | 客户业务管理员、品牌/工厂管理员 |
| 典型操作 | License 导入、服务监控、备份恢复、版本升级 | 内部租户、组织、账号、业务权限、额度 |
| 风险等级 | 高，可能影响整套系统运行 | 中，主要影响业务配置和数据可见范围 |

OP 本地化的基本关系如下：

```text
厂商授权/License 中心
  └─ 生成授权文件、模块许可、有效期、额度

客户本地 OP 实例
  ├─ 系统管理控制台（面向客户 IT / 运维 / 厂商交付）
  │    └─ License、服务监控、系统日志审计、备份恢复、版本升级
  └─ 客户实例管理后台（面向客户业务管理员）
       └─ 内部租户、组织、品牌、工厂、供应商、账号、业务权限、额度
```

#### 3.2.4 混合云扩展预留

混合云暂不作为前期标准部署方式。后期如客户存在“云端统一管理 + 本地工厂/设备/数据接入”的要求，可通过新增 Hybrid Profile 扩展。

推荐的混合云形态是“云端 IMP + 本地边缘节点”：

```text
厂商云端 IMP
  ├─ 租户管理
  ├─ 发码管理
  ├─ 码库与批次管理
  ├─ 统计与风控
  └─ API 网关

客户本地边缘节点
  ├─ 工厂/设备接入
  ├─ 码包下载缓存
  ├─ 采集数据暂存
  ├─ 断网续传
  └─ 本地日志
```

##### 混合云预留能力

| 能力 | 设计要求 |
| --- | --- |
| 部署 Profile 扩展 | Profile 不写死为 SaaS/OP，后续可新增 Hybrid Profile |
| 数据分层 | 区分云端统一数据、本地缓存数据、本地留存数据 |
| 接口网关 | 支持云端调用、本地节点调用和第三方系统调用 |
| 异步同步 | 支持包材回传、采集数据、激活数据的断点续传和重试 |
| 节点管理 | 后续可管理本地节点状态、版本、授权、心跳、日志 |
| 授权模型 | License/授权模型支持客户实例授权，也能扩展到本地边缘节点授权 |

如果混合云只是“云端主系统 + 本地轻量边缘节点”，改动可控；如果要求云端和本地都能独立承载核心业务，则会涉及数据一致性、冲突处理、双端升级、双端审计和运维责任划分，改动较大，需要单独立项。

### 3.3 IMP 与外部系统关系

> 图：IMP 与外部系统关系图

| 系统类型 | 典型系统 | 关系类型 | 交互方式 |
| --- | --- | --- | --- |
| 管理类系统 | ERP/WMS/MES/数据中台 | 平行关系 | 管理集成 API（主数据同步、批次状态、运营数据） |
| 生产类系统 | 赋码系统/采集关联系统 | 生态协同关系 | 生产采集类 API（码包下载、关系上传、校验回传） |
| 业务类系统 | 防窜物流系统/智能营销系统/互动营销系统/追溯系统/防伪系统/BDE 等 | 外部消费关系 | 业务/查询类 API（标识解析、关系查询、状态校验、扫码事件回传、业务路由） |

#### IMP 与第三方业务系统的边界原则

- **IMP 管标识事实**：负责码、码包、码关系、码状态、批次、扫码解析、风控判断、业务路由和标识事件。
- **业务系统管业务玩法**：防窜物流、智能营销、互动营销、追溯、防伪等系统负责具体业务流程、活动规则、页面承载、渠道规则和业务闭环。
- **多系统并存**：同一企业可同时接入多套第三方业务系统，IMP 通过应用凭据、业务场景、品牌/SKU、码类型、路由规则区分调用方和目标系统。
- **强边界集成**：BDE 只是第三方业务系统之一，不能在 IMP 内置专属耦合逻辑；如需对接，通过标准 API、事件回调和路由配置完成。

## 第4章 标识业务核心模型

本章定义 IMP 作为一物一码标识中台必须具备的业务底座。后续产品功能、API、风控和统计均应基于本章模型展开。

### 4.1 核心主数据关系模型

IMP 的主数据不是普通基础档案，而是发码、赋码、批次激活、扫码解析和业务路由的共同依据。

```text
企业/租户
  └─ 子公司 / 事业部 / 品牌
       └─ 产品 / SKU
            └─ 包装层级 / 码类型 / 码制规则
                 └─ 发码模版
                      └─ 发码订单
                           └─ 码包
                                └─ 包材赋码 / 采集回传
                                     └─ 成品批次 / 激活策略
                                          └─ 扫码解析 / 第三方业务系统调用
```

| 主数据对象 | 业务含义 | 关键关系 |
| --- | --- | --- |
| 企业/租户 | IMP 的客户主体或客户内部管理主体 | 关联子公司、事业部、品牌、账号、额度 |
| 子公司/事业部/品牌 | 集团内部经营单元 | 关联产品、SKU、业务系统路由规则 |
| 产品/SKU | 赋码和扫码解析的商品主体 | 关联包装层级、发码模版、成品批次 |
| 包装层级 | 瓶、盒、箱、垛等包装结构 | 决定单码、同级码、层级码关系 |
| 供应商/包材厂 | 码包接收和包材赋码主体 | 关联发码订单、码包下载权限、回传责任 |
| 码制规则/发码模版 | 码生成和业务属性规则 | 关联码类型、扩展字段、审批流程、排废率 |
| 第三方业务系统 | 消费 IMP 标识能力的业务系统 | 关联应用凭据、API 权限、路由规则、回调地址 |

### 4.2 码生命周期状态模型

码生命周期状态模型用于统一判断码是否可用、是否可扫码、是否可对外路由、是否存在风险。

| 状态 | 含义 | 典型触发动作 | 状态处理原则 |
| --- | --- | --- | --- |
| 已生成 | 码已由发码引擎生成并进入码库 | 发码申请审核通过并生成码包 | 可进入分配或作废 |
| 已分配 | 码已归属到发码订单或码包 | 码包生成、码段分配 | 记录订单和码包归属 |
| 已下载 | 码包已被包材厂或授权对象下载 | 码包下载成功 | 记录下载人、次数、时间、IP |
| 已赋码 | 码已用于包材印刷或赋码生产 | 包材厂赋码完成并提交回传 | 需等待回传校验 |
| 已回传 | 赋码或采集数据已回传 IMP | 文件上传、接口回传、设备采集 | 进入校验和签收流程 |
| 已签收 | 品牌方确认包材接收或验收 | 包材签收确认 | 可进入成品绑定或批次激活 |
| 已激活 | 码与商品/批次绑定后，对外扫码解析与业务系统调用正式生效 | 批次激活、区间激活、入库/出库激活、外部系统回调激活 | 可对外查询和业务路由 |
| 已查询 | 码发生过扫码查询或系统查询 | 消费者扫码、业务系统查询、稽查查询 | 通常作为事件流水，不改变主状态 |
| 已冻结 | 码因风险、争议或人工处置暂时不可用 | 风控触发、人工冻结 | 可解冻或转作废 |
| 已作废 | 码被确认失效，不再允许业务使用 | 批量作废、问题码作废、过期作废 | 原则上不可逆 |
| 已过期 | 码超过有效期或业务活动期 | 有效期到期、活动结束 | 可配置是否自动作废 |
| 异常待处理 | 码状态、关系或扫码行为存在异常 | 回传失败、关系冲突、异常扫码 | 进入异常闭环 |

#### 状态机原则

- 码状态变更必须留痕，记录触发来源、操作人、系统来源、时间和变更前后状态。
- 已作废状态原则上不可逆；冻结、异常待处理可经授权流程恢复。
- 查询、扫码、风控命中通常写入事件流水，不应随意改变码主状态。
- 不同企业可配置激活节点，但底层“已激活”语义必须统一为“可对外解析与业务调用正式生效”。

### 4.3 码关系模型

码关系模型用于支撑防伪、溯源、召回、渠道稽查、扫码路由和业务系统调用。

| 关系类型 | 含义 | 建立时机 | 校验重点 |
| --- | --- | --- | --- |
| 单码实体 | 单个码作为独立标识对象 | 发码生成时 | 唯一性、状态、归属 |
| 同级关系 | 同一包装层级多个码之间的绑定关系，如内外码、正反码 | 模版生成、包材回传、采集上传 | 一一对应关系、重复绑定、缺失关系 |
| 层级关系 | 瓶、盒、箱、垛等不同层级间的父子关系 | 产线采集、设备上传、文件导入 | 父子唯一性、层级完整性、跨批次异常 |
| 外部码映射 | 第三方码或历史码与 IMP 码之间的映射关系 | 第三方码导入、历史数据迁移 | 来源可信度、重复映射、解析优先级 |

#### 关系治理原则

- 关系数据应版本化保存，原则上不直接覆盖历史关系。
- 关系来源需记录为系统生成、设备采集、文件导入、第三方导入或人工修正。
- 对关系数据设置可信等级，系统生成和设备采集优先级高于人工导入。
- 同级码和层级码关系不仅服务查询，也服务防窜、追溯、召回、风控和扫码路由。

### 4.4 发码订单、码包、赋码批次、成品批次关系

IMP 需要区分“申请了多少码”“交付了哪些码”“实际用了哪些码”“哪些码具备业务身份”四类事实。

| 对象 | 回答的问题 | 关键数据 | 与其他对象关系 |
| --- | --- | --- | --- |
| 发码订单 | 谁申请、申请多少、用于什么产品和场景 | 产品/SKU、码类型、申请数量、排废率、审批状态 | 生成一个或多个码包 |
| 码包 | 哪些码被打包交付给谁 | 码段/码文件、下载权限、密码、有效期、下载记录 | 归属发码订单，可被包材厂下载 |
| 赋码批次 | 实际赋码生产和回传情况 | 回传数量、损耗数量、排废数量、关系数据、异常数据 | 由包材厂或设备回传，校验后进入签收 |
| 成品批次 | 哪些码与商品批次正式绑定 | 商品/SKU、生产批次、入库/出货信息、激活策略 | 激活后支持扫码解析和业务调用 |

### 4.5 包材厂协同与回传校验规则

包材厂是 IMP 落地中的关键协同方，系统需要用规则降低人工沟通和异常处理成本。

| 环节 | 规则要求 | 异常处理 |
| --- | --- | --- |
| 包材厂账号授权 | 按供应商、品牌、产品、订单授权访问范围 | 未授权不可见码包 |
| 码包下载 | 支持有效期、下载次数、IP 白名单、密码短信或独立密码 | 超期、超次、异常 IP 需拦截或告警 |
| 文件格式 | 统一码包文件格式、命名规则、加密方式、字段说明 | 格式不符不可导入或需人工确认 |
| 补码/重下/撤销 | 支持补码申请、码包重下、授权撤销和作废联动 | 重下需记录原因和审批 |
| 赋码损耗与排废 | 区分申请量、发码量、下载量、实际赋码量、排废量 | 排废异常进入异常闭环 |
| 回传校验 | 校验订单、码包、码数量、码关系、重复码、缺失码 | 校验失败可退回、部分通过或人工兜底 |
| 人工兜底 | 对不具备系统能力的包材厂提供网页上传和模板导入 | 所有人工处理必须保留审计记录 |

### 4.6 扫码解析、状态校验与业务路由模型

扫码解析与业务路由是 IMP 的“查码”核心能力，不应从属于某一套业务系统。

扫码请求进入 IMP 后，推荐处理链路如下：

```text
扫码请求
  → 码解析
  → 码状态校验
  → 码关系查询
  → 风险判定
  → 业务路由匹配
  → 第三方业务系统跳转/API调用
  → 扫码事件记录
```

| 能力 | 功能说明 |
| --- | --- |
| 码解析 | 识别码值、码类型、码制规则、所属租户/品牌/SKU |
| 状态校验 | 判断码是否已激活、冻结、作废、过期或异常待处理 |
| 关系查询 | 查询同级关系、层级关系、外部码映射关系 |
| 风险判定 | 判断重复扫码、高频扫码、跨区域扫码、作废码访问等异常 |
| 业务路由 | 按租户、品牌、SKU、码类型、场景、活动、第三方系统配置进行路由 |
| 事件记录 | 记录扫码时间、地点、设备、IP、调用方、路由结果和风控结果 |

#### 多第三方业务系统接入原则

- 每套第三方业务系统应有独立应用身份、API 权限、回调地址和路由规则。
- 同一企业可以同时接入多套防窜物流、智能营销、互动营销、追溯、防伪系统。
- IMP 不承载具体营销玩法、经销商业务流程或活动页面，只负责标识解析、状态判断、风险判断、路由分发和事件沉淀。
- 当目标业务系统不可用时，应支持降级策略，例如默认页、错误页、备用路由或仅记录扫码事件。

### 4.7 异常码、作废码、冻结码和风险码处理机制

IMP 需要区分不同异常类型，避免把所有问题都混入“异常码”。

| 类型 | 定义 | 典型处理 |
| --- | --- | --- |
| 异常码 | 状态、关系、回传或扫码行为存在异常，需要进一步判断 | 进入异常池、分派处理、补充校验 |
| 作废码 | 已确认不可继续使用的码 | 禁止解析、禁止路由、保留历史记录 |
| 冻结码 | 暂停使用但尚未最终确认的码 | 暂停解析或限制路由，可经审批解冻 |
| 风险码 | 被风控规则命中的码或扫码事件 | 记录风险等级，触发告警、拦截或人工复核 |

异常闭环应包括：发现来源、异常类型、影响范围、处理人、处理动作、处理结果、恢复状态和审计记录。

| 采购人员 | 品牌商负责包材采购与签收的人员 | 商户移动端 |

### 5.2 访问层产品定义

#### 5.2.1 商户后台（品牌商端）

**功能定位**：企业管理标识全生命周期的自服务操作平台，为企业提供从发码规划、码包生成、赋码协同到成品管理的完整业务闭环能力。

**核心能力**：

- 发码模版配置与发码申请管理
- 码包授权分发与下载状态监控
- 码包回传与关系数据管理
- 成品批次管理与激活
- 数据统计与风控管理
- 基础数据与权限配置
- 第三方业务系统接入配置与扫码路由管理

#### 5.2.2 商户移动端（品牌商端）

**功能定位**：企业移动办公场景的轻量化操作入口，支持现场人员完成包材签收确认、业务审批等移动化操作。

**核心能力**：

- 包材签收确认（采购/仓库人员现场签收）
- 发码申请审批（审批人移动端处理）
- 消息通知与待办提醒
- 关键数据看板（发码/激活进度概览）

#### 5.2.3 包材下载门户（包材厂端）

**功能定位**：包材厂操作人员使用的码包接收平台，通过生态账号登录后可完成发码申请、码包下载和数据回传操作。

**核心能力**：

- 发码申请提交（包材厂自主发起发码申请）
- 待下载码包列表查看与筛选
- 码包下载与解压密码接收
- 赋码数据回传（同级码关系/单码）
- 下载与回传记录查询

#### 5.2.4 开放 API 网关（系统集成入口）

**功能定位**：为包材厂、赋码设备、生产系统、管理系统和多套第三方业务系统提供标准化 API 接入能力，支撑生产采集、业务消费、管理集成、扫码解析和事件回传等场景。

##### API 分类

| API 类型 | 服务对象 | 核心能力 |
| --- | --- | --- |
| 标识生成类 API | 业务系统、运营后台 | 生码申请、发码订单查询、码包状态查询 |
| 生产采集类 API | 工厂产线、赋码设备、包材厂系统 | 码包下载、关系上传、码包回传、回传校验 |
| 管理集成类 API | ERP/WMS/MES/数据中台 | 主数据同步、批次状态同步、入库/出货事件同步 |
| 业务解析类 API | 防窜物流、智能营销、互动营销、追溯、防伪、BDE 等第三方业务系统 | 码解析、状态查询、关系查询、风控结果查询、业务路由 |
| 事件回传类 API | 第三方业务系统 | 扫码事件回传、业务事件回传、异常结果回传 |

##### API 治理要求

- 每套第三方业务系统独立分配应用身份、密钥、权限范围和回调地址。
- 核心写入类接口需支持幂等键，避免重复发码、重复回传、重复激活。
- 接口调用需支持签名鉴权、时间戳、防重放、限流、审计和错误码规范。
- 回调类接口需支持重试、失败记录、人工补偿和状态对账。

### 5.3 业务层产品定义

#### 5.3.1 发码模版

**功能定位**：预定义的标识生成规则和配置模板，包含编码规则、码类型、扩展字段、层级结构、审批流程等配置信息。

##### 模版类型

| 模版类型 | 适用场景 | 典型案例 |
| --- | --- | --- |
| 单码模版 | 仅需单一赋码的简单场景 | 瓶盖内码营销、单品防伪码 |
| 同级码模版 | 同一包装层级需内外码配合 | 盖内码+盖外码、标签正反码 |
| 层级码模版 | 多级包装需建立父子关系 | 瓶→盒→箱→垛的层级追溯 |

**配置维度**：编码规则、码类型、扩展字段、产品/SKU、包装层级、码关系类型、审批流程、排废率、默认激活策略、默认业务路由策略

#### 5.3.2 发码申请

**功能定位**：标识生成的执行模块，基于已配置的发码模版，完成发码申请提交和码包审核管理。

##### 子功能清单

| 子功能 | 功能说明 | 关键操作 |
| --- | --- | --- |
| 发码申请 | 选择发码模版，填写码包名称、申请数量、排废率、码包文件数等信息，系统自动计算实际发码数量，提交发码需求（SaaS 公有云版提交时冻结发码券额度，审核通过后正式扣减） | 新建申请、提交审核 |
| 码包审核 | 审批发码申请，根据发码模版配置的审批流程进行审核 | 审核通过、审核驳回 |

- **核心流程**：选择模版 → 填写申请信息 → 提交审批 → 审批通知
- **审批模式**：无需审批、单人审批、或签审批、会签审批

#### 5.3.3 赋码管理

**功能定位**：品牌商侧的赋码协作管理中心，负责码包下载、授权分发、多端下载状态监控及关联数据的回传管理。

##### 子功能清单

| 子功能 | 功能说明 | 关键操作 |
| --- | --- | --- |
| 码包下载 | 管理码包的授权分发，监控各端下载状态，支持账号登录+密码短信安全机制 | 下载授权、安全参数配置、状态监控、授权撤销 |
| 码包回传 | 接收包材厂赋码后回传的数据，支持单码、同级码关系、层级码关系回传，需关联发码订单、码包和赋码批次 | 选择发码订单、文件上传、数据校验、赋码采集率统计 |
| 包材签收 | 品牌商确认包材接收，采购/仓库人员现场验收确认 | 签收确认、验收记录 |

##### 核心能力

- **码包下载**：商户侧网页端直接下载码包，配套赋码系统同步支持（后续联动赋码系统，无赋码系统的包材厂也可通过网页适配）
- **码包下载管理**：下载授权、有效期控制、下载次数控制、IP 白名单、下载状态监控
- **码包回传管理**：单码回传、同级码关系回传、层级码关系回传、排废数据回传、数据校验、赋码采集率统计
- **异常处理**：支持回传失败退回、部分通过、重复码拦截、缺失码提示、人工兜底导入和异常闭环

#### 5.3.4 成品管理

**功能定位**：成品管理是标识中台将标识与商品、批次、包装层级和业务属性绑定的关键环节。成品激活的统一语义是：码与商品/批次绑定后，对外扫码解析与业务系统调用正式生效。

##### 子功能清单

| 子功能 | 功能说明 | 关键操作 |
| --- | --- | --- |
| 成品批次管理 | 创建、激活、管理成品批次，绑定商品、批次和包装信息 | 创建批次、批次激活、区间激活、入库激活、出货激活 |
| 激活策略配置 | 配置不同企业、品牌、SKU、业务场景下的激活节点 | 按批次激活、按区间激活、按上传文件激活、按入库单激活、按出货单激活、按外部系统回调激活 |
| 商品查询 | 根据码查询商品信息和关联关系 | 精确查询 |
| 二维码作废 | 对问题码或过期码进行作废处理 | 单码作废、批量作废、作废原因记录 |

##### 5 种批次来源类型

- 纯单码批次
- 前关联组合码批次
- 后关联同级码批次
- 后关联层级码批次
- 第三方码导入批次

**核心操作**：批次创建、商品绑定、激活策略配置、批次激活、区间激活、二维码作废

#### 5.3.5 扫码解析与业务路由

**功能定位**：扫码解析与业务路由是 IMP 的统一查码入口，负责对扫码请求进行码解析、状态校验、关系查询、风控判断、业务路由和事件记录。

##### 子功能清单

| 子功能 | 功能说明 | 关键操作 |
| --- | --- | --- |
| 码解析 | 识别码值、码类型、码制规则、所属租户/品牌/SKU | 码值解析、规则匹配 |
| 状态校验 | 判断码是否已激活、冻结、作废、过期或异常 | 状态查询、拦截判断 |
| 关系查询 | 查询同级码、层级码、第三方码映射关系 | 关系追溯、父子查询 |
| 业务路由 | 按租户、品牌、SKU、码类型、场景、第三方系统配置进行路由 | 路由规则配置、跳转、API 调用 |
| 扫码事件 | 记录扫码时间、地点、设备、IP、调用方、路由结果 | 事件沉淀、统计分析 |

**边界说明**：IMP 不承载具体营销玩法、经销商流程或活动页面；第三方业务系统负责具体业务闭环，IMP 负责标识解析、状态判断、风险判断、路由分发和扫码事件沉淀。

#### 5.3.6 数据统计

**功能定位**：标识中台的运营数据可视化模块，提供发码、激活、码包使用等标识层面的统计分析能力。

##### 统计维度

- **生产协同指标**：发码量、下载量、回传量、排废量、采集率、回传及时率、异常回传率。
- **标识运营指标**：激活量、成品批次数、作废量、冻结量、异常码数量。
- **业务应用指标**：扫码量、首次扫码率、重复扫码率、区域分布、第三方系统调用量、业务路由命中情况。
- **风险指标**：未激活扫码、作废码访问、跨区域扫码、高频扫码、疑似撞库、码关系异常。

#### 5.3.7 风控管理

**功能定位**：标识中台的一物一码风险识别与处置模块，识别并防范对标识服务的异常访问、异常关系、异常状态和异常业务调用，保护标识数据资产安全。

##### 典型风险场景

- 未激活扫码、作废码访问、冻结码访问、过期码访问。
- 同一码短时高频扫码、跨区域扫码、异常设备扫码、异常 IP 访问。
- 同级码/层级码关系不一致、父子关系缺失、跨批次关系异常。
- 批量撞库、伪造码访问、非授权第三方系统调用。

**处置方式**：仅记录、提示用户、进入人工审核、拦截访问、冻结码、作废码、触发预警、通知第三方业务系统。

#### 5.3.8 基础管理

**功能定位**：基础数据与系统配置中心，负责维护企业主数据、数据字典、账号权限、业务预警和操作审计。

##### 核心能力

- 基础设置（企业、子公司、事业部、品牌、产品/SKU、包装层级、工厂、供应商/包材厂档案）
- 数据字典管理
- 账号与权限管理（RBAC 权限体系）
- 第三方业务系统注册、应用凭据、API 权限和路由规则配置
- 预警设置
- 操作日志审计

### 5.4 服务层说明

服务层（标识中心、订单中心、批次中心、配置中心、主数据中心、解析路由中心）属于技术实现层，为业务层提供统一的能力支撑，本文档不展开详述。

### 5.5 平台层产品定义

平台层是 IMP 的部署与运营基础能力，负责承接不同部署 Profile 下的差异化能力。核心原则是：业务层和服务层尽量统一，部署差异集中在平台层处理；Profile 设计需支持后续扩展 Hybrid Profile。

#### 5.5.1 SaaS 公有云平台能力

**功能定位**：面向厂商级多租户场景的账户、订阅、计费和统一运营能力。

| 子功能 | 功能说明 |
| --- | --- |
| 租户开户 | 厂商为外部品牌商客户创建租户，配置套餐、有效期和初始额度 |
| 订阅续费 | 服务订阅、续费、停用和恢复管理 |
| 发码券管理 | 发码券余额查询、配额预警提醒、额度冻结与扣减 |
| 消费明细 | 发码消耗记录、账单导出下载 |
| 发码券购买 | 在线充值入口、套餐包选购；前期也可支持人工充值或合同充值 |
| 统一升级 | 厂商统一发布版本和维护平台环境 |

#### 5.5.2 专属云/私有云实例管理能力

**功能定位**：面向单客户独立实例场景，管理客户实例、实例授权、内部租户和客户级隔离能力。

| 子功能 | 功能说明 |
| --- | --- |
| 实例信息管理 | 记录客户实例、部署环境、版本、资源规格、交付状态 |
| 实例级授权 | 按合同配置模块、有效期、码量额度、并发数等授权范围 |
| 内部租户管理 | 支持集团内部品牌、事业部、子公司、工厂等内部租户管理 |
| 内部额度分配 | 将客户购买的总额度分配给内部租户、品牌或业务单元 |
| 独立升级 | 按客户约定窗口进行实例升级、灰度、回滚 |
| 托管运维 | 按合同约定支持厂商托管或双方协作运维 |

#### 5.5.3 OP 本地化系统管理与 License 能力

**功能定位**：面向客户本地化部署场景，提供本地系统控制、离线授权、审计、备份、升级和客户自主运维能力。

| 子功能 | 功能说明 |
| --- | --- |
| License 导入与校验 | 支持 License 文件、离线激活、硬件指纹绑定、授权有效期和模块开关 |
| 服务监控中心 | 系统健康度监控、性能看板、告警管理 |
| 系统日志审计 | 登录日志、系统操作日志、异常日志、授权变更日志，满足等保和内控要求 |
| 数据备份与恢复 | 备份策略配置、备份执行、恢复和升级前备份 |
| 版本升级 | 上传升级包、升级前检查、执行升级、回滚 |
| 本地初始化 | 初始化企业、实例标识、超级管理员和基础系统参数 |

#### 5.5.4 统一租户与权限模型

**功能定位**：在统一产品底座下，兼容 SaaS 公有云的厂商级多租户，以及专属云/私有云、OP 本地化的客户内部多租户。

| 模型 | 说明 |
| --- | --- |
| 统一租户模型 | 保留 tenant 概念，支持厂商级租户和客户内部租户两种口径 |
| 统一组织模型 | 支持集团、品牌、事业部、工厂、供应商等组织层级 |
| 统一权限模型 | 支持账号、角色、菜单、数据权限、操作权限，并区分系统运维权限与业务管理权限 |
| 统一额度模型 | 支持发码券、码量额度、模块额度、租户内部分配 |
| 统一模块开关 | 支持按版本、套餐、License 控制功能模块 |
| 统一审计模型 | 操作日志、登录日志、授权变更、数据变更留痕 |
| 统一应用接入模型 | 支持多套第三方业务系统的应用身份、API 权限、回调地址、路由规则和调用审计 |

#### 5.5.5 混合云扩展预留能力

**功能定位**：为后续混合云部署保留能力边界。平台层设计应避免把部署方式写死，以便后续按客户场景扩展本地边缘节点、数据同步和节点授权能力。

| 能力 | 功能说明 |
| --- | --- |
| Hybrid Profile | 后续新增混合云部署 Profile，启用云端主系统与本地边缘节点协同能力 |
| 本地节点管理 | 管理本地节点状态、版本、授权、心跳、日志和连接状态 |
| 数据同步机制 | 支持采集数据、回传数据、激活数据的异步同步、断点续传和失败重试 |
| 本地缓存策略 | 支持码包下载缓存、采集数据暂存和离线期间的数据保护 |
| 节点级授权 | 将授权模型从客户实例扩展到本地边缘节点 |
| 节点监控与审计 | 支持本地节点运行日志、同步日志、异常日志和运维审计 |

### 5.6 治理层说明

治理层是贯穿全流程的横切能力，包括身份权限、审计追责、安全管控、数据质量、异常闭环、接口治理、运营度量等。具体功能已融入第4章标识业务核心模型、5.3.7 风控管理与 5.3.8 基础管理模块。

## 第6章 建设策略

本章明确标识中台建设的核心策略原则，指导第7章能力演进路线的阶段划分。

### 6.1 先标识事实源，后业务系统集成

**策略说明**：先完成 IMP 作为标识事实源的独立运行能力，再开放生态对接与第三方业务系统集成。

**实施要点**：

- 优先打通标识主链路（发码→码包→赋码→回传→激活→扫码解析）
- IMP 优先实现独立闭环，不依赖 BDE 或任一具体业务系统即可交付使用
- 核心能力验证后，再开放 API 网关、扫码路由和事件回传能力
- 第三方业务系统集成作为生态扩展阶段，BDE 只是其中一种接入对象
- 包材下载门户、赋码系统、第三方业务系统 API 分阶段开放

### 6.2 先标准云化闭环，后扩展专属/私有云与 OP 本地化

**策略说明**：先完成标准云化产品闭环，再沉淀专属云/私有云和 OP 本地化所需的平台能力、授权能力和运维能力。

**实施要点**：

- SaaS 公有云适合作为标准产品能力验证阵地和优先商用形态
- SaaS 公有云验证通过后，再抽取专属云/私有云的实例管理、内部多租户和客户级隔离能力
- OP 本地化以 License、系统管理控制台、日志审计、备份恢复、离线升级等能力为核心，适合作为独立部署专项能力建设
- 三类标准部署方式共享业务层和服务层能力，部署差异集中在平台层通过 Profile 适配

### 6.3 先核心模型，后扩展业务场景

**策略说明**：先完成码生命周期、码关系、码包、批次、扫码解析等核心模型，再扩展具体行业场景与补充能力。

**实施要点**：

- 主干链路（前置准备→申请码包→包材赋码→包材签收→成品入库→扫码解析）优先闭环
- 主数据先支持核心项（组织/品牌/SKU/包装层级/工厂/供应商/第三方业务系统），后续扩展更多业务属性
- 码生命周期、码关系、激活策略、扫码路由应作为底层模型优先固化
- 边缘场景（如复杂审批、特殊编码规则、层级采集）后续迭代补充

### 6.4 先标准交付，后部署形态扩展

**策略说明**：先形成标准产品闭环和可复用产品底座，再按客户类型、数据主权要求和运维边界扩展专属云/私有云、OP 本地化与混合云能力。

**实施要点**：

- 标准产品闭环应优先保证可演示、可试点、可售卖
- 专属云/私有云需要补充实例管理、独立升级、客户内部多租户和客户级隔离能力
- OP 本地化需要补充 License、系统管理控制台、日志审计、备份恢复、离线升级和本地运维手册
- 混合云后续通过 Hybrid Profile、本地边缘节点、接口网关和异步数据同步机制扩展，不作为前期标准部署方式
- 对外售卖时可以保留专属云/私有云、OP 本地化口径，但交付承诺应单独评估资源、周期和客户环境

### 6.5 先强边界，后深集成

**策略说明**：IMP 与第三方业务系统按强边界集成，先通过标准 API、应用凭据、路由规则和事件机制完成解耦，再根据客户需求做深度适配。

**实施要点**：

- IMP 不内置具体营销玩法、防窜流程、经销商流程或活动页面
- 第三方业务系统负责业务玩法和业务闭环，IMP 负责标识事实、解析、状态、关系、风控、路由和事件
- 同一企业可同时接入多套第三方业务系统，每套系统独立配置应用身份、API 权限、回调地址和路由规则
- BDE 与 IMP 按第三方业务系统关系处理，不作为 IMP 的内置模块或默认依赖

## 第7章 能力演进路线

本章从产品蓝图视角描述标识中台的能力演进路线，与第6章建设策略对应，覆盖第4章核心模型与第5章产品功能。本章不是具体项目排期，具体版本范围应在专项项目计划中定义。

## 第8章 决策与 Q&A 清单

### 8.1 已决策事项

| 问题 | 结论 | 说明 |
| --- | --- | --- |
| IMP 的部署方式如何统一表述？ | 标准部署口径统一收敛为三类：SaaS 公有云、专属云/私有云、OP 本地化。 | 混合云暂不作为前期标准部署方式，但通过 Hybrid Profile、本地边缘节点和数据同步机制预留。 |
| SaaS 公有云之外，其他部署方式是否支持多租户？ | 支持，但含义不同。 | SaaS 公有云是厂商级多租户；专属云/私有云和 OP 本地化是客户独立实例内的客户内部多租户。 |
| 多租户与部署方式是什么关系？ | 两者是两个维度。 | 部署方式回答系统部署在哪里；多租户回答一套系统如何服务多个组织。 |
| SaaS 公有云的租户开户、订购、续费在哪里管理？ | 在厂商运营后台（米多星球）管理。 | 厂商负责客户租户、套餐、订阅周期、发码券、续费、停用等。 |
| 专属云/私有云的内部租户管理后台在哪里？ | 在客户独立实例内。 | 厂商运营后台（米多星球）管理合同、版本、模块授权和服务状态；客户实例管理后台管理内部组织、账号、权限和额度。 |
| OP 本地化为什么要区分系统管理控制台和客户实例管理后台？ | 两者管理对象、使用者和风险等级不同。 | 系统管理控制台管 License、服务、备份、升级；客户实例管理后台管内部租户、组织、账号、权限和业务额度。 |
| 成品激活的业务语义是什么？ | 码与商品/批次绑定后，对外扫码解析与业务系统调用正式生效。 | 激活节点可按生产完成、入库、出货、活动开始或外部系统回调配置，但底层语义保持一致。 |
| 扫码解析与扫码路由是否属于 IMP 核心能力？ | 属于。 | 扫码解析是“查码”核心能力，负责码解析、状态校验、关系查询、风控判定、业务路由和扫码事件沉淀。 |
| IMP 与 BDE 的关系是什么？ | BDE 是第三方业务系统之一，不是 IMP 内置模块。 | IMP 与 BDE、防窜、营销、追溯、防伪等系统通过标准 API、事件回调和路由配置集成。 |
| IMP 是否支持同一企业对接多套第三方业务系统？ | 必须支持。 | 同一企业可能存在多子公司、多品牌、多事业部、多 SKU，并采购多套业务系统；IMP 作为标识中台应统一提供多系统接入能力。 |
| 标识中台与其他外围系统之间的关系？ | ERP/WMS/MES 是管理集成关系；赋码/采集系统是生产协同关系；业务系统是标识能力消费关系。 | 不同系统通过管理集成 API、生产采集类 API、业务解析类 API 和事件回传 API 连接。 |
| 是否需要在同一阶段同时完整建设三种部署方式？ | 不建议。 | 整体蓝图统一规划三种部署方式，但实际建设应分阶段推进。 |
| 后期如果支持混合云部署，当前结构是否支撑？ | 当前结构可支撑轻量混合云。 | 若采用“云端 IMP + 本地轻量边缘节点”，改动可控；若要求云端和本地都独立承载核心业务，则需专项设计。 |
| 所有包材厂人员能否按规范完成按单下载码包、按单回传码包数据？ | 这是产品落地阶段重点验证风险。 | 通过包材下载门户和赋码系统两个入口覆盖不同成熟度的包材厂，并验证下载授权、文件格式、回传校验和异常处理。 |
