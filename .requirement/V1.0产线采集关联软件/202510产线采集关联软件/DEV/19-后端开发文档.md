# 后端开发文档（无代码版）

## 1. 概述

本文档描述产线采集关联软件的后端实现，包括业务逻辑、数据库操作、接口设计等。后端采用Spring Boot框架，使用MyBatis-Plus进行数据库操作，遵循领域驱动设计（DDD）架构。

## 2. 技术架构

### 2.1 技术栈
- **框架**: Spring Boot
- **ORM**: MyBatis-Plus
- **数据库**: SQL Server
- **架构模式**: 领域驱动设计（DDD）
- **事务管理**: Spring @Transactional

### 2.2 项目结构
- **miduo-common**: 公共模块（DTO、工具类）
- **miduo-entity**: 实体模块（VO、Request、Response）
- **miduo-domain**: 领域模块（Domain Model、Repository接口）
- **miduo-infrastructure**: 基础设施模块（Repository实现、Mapper、PO）
- **miduo-application**: 应用服务模块（Application Service）
- **miduo-controller**: 控制器模块（REST API）
- **miduo-frontend**: 前端模块（JavaFX UI）
- **miduo-bootstrap**: 启动模块

## 3. 核心业务逻辑

### 3.1 任务管理

#### 3.1.1 添加任务
**业务逻辑**:
1. 先操作`ProductionOrder`表：
   - 设置`OrderStatus=1`（生产中）
   - 设置`ProductSourceType=2`（产线）
   - 设置`ProductionLine=001`（暂定）
   - 生成订单号
2. 再操作`ProductionOrderDetail`表：
   - 插入任务明细记录
   - 设置`IsDel=0`（未删除）
   - 设置`Type`（1-有箱码，2-无箱码）

**数据库操作要点**:
- 先插入`ProductionOrder`表记录，获取生成的订单号
- 使用生成的订单号插入`ProductionOrderDetail`表记录
- 所有字段必须设置默认值，字符串字段如果为空设置为空字符串

#### 3.1.2 更新任务
**业务逻辑**:
- 更新`ProductionOrderDetail`表的字段
- 不修改`IsDel`字段（逻辑删除由删除接口处理）

**数据库操作要点**:
- 根据ID查询现有记录
- 更新需要修改的字段
- 保持`IsDel`字段不变

#### 3.1.3 删除任务
**业务逻辑**:
- 逻辑删除：设置`ProductionOrderDetail.IsDel=1`
- 不删除`ProductionOrder`表记录

**数据库操作要点**:
- 使用UpdateWrapper更新`IsDel`字段为1
- 不执行物理删除操作

#### 3.1.4 查询任务
**业务逻辑**:
1. 从`ProductionOrderDetail`表查询任务明细（`IsDel=0`）
2. 关联查询`ProductionOrder`表获取订单状态
3. 支持模糊查询：订单号、批次号、产品名称
4. 支持状态筛选：`OrderStatus`（0-待生产，1-生产中，2-已完成，5-提前结单）
5. 完成数量：从`ProductionOrderDetail.OrderCount`字段获取

**数据库操作要点**:
- 使用分页查询，支持大量数据
- 查询条件使用LIKE进行模糊匹配
- 关联查询`ProductionOrder`表获取订单状态信息
- 按ID降序排列，最新记录在前

#### 3.1.5 更新任务状态
**业务逻辑**:
1. 更新`ProductionOrder.OrderStatus`字段
2. 状态值：0-待生产，1-生产中，2-已完成，5-提前结单
3. 启用任务时（status=1）需要校验采集模式切换：
   - 查询`CodeRelationUpload`表最新记录（按`AddTime`降序，取第一条）
   - 检查`Type`是否一致
   - 如果不一致，检查`BigSerialNumber`是否有值
   - 如果`BigSerialNumber`为空，提示无法切换模式

**数据库操作要点**:
- 使用分页查询获取最新记录（兼容SQL Server）
- 更新`ProductionOrder`表的`OrderStatus`字段
- 校验逻辑需要查询`CodeRelationUpload`表

### 3.2 码采集与关联

#### 3.2.1 有箱码采集
**业务逻辑**:
1. 接收箱码（`SmallSerialNumber`）
2. 检查重复码：
   - 查询`CodeRelationUpload`表的`SmallSerialNumber`和`BigSerialNumber`字段
   - 如果存在重复，返回重复提示
3. 生成或获取`TagNo`（标签编号）：
   - 使用`ConcurrentHashMap`缓存当前订单的`TagNo`
   - 如果不存在，生成新的`TagNo`
4. 记录采集数量：
   - 使用`ConcurrentHashMap`缓存当前`TagNo`的采集数量
   - 每次采集后数量+1
5. 插入`CodeRelationUpload`表：
   - 设置`Type=1`（有箱码）
   - 设置`Status=0`（未完成）
   - 设置`IsUpload=1`（不同步）
   - 设置`IsDel=0`（未删除）
6. 满垛判断：
   - 当采集数量达到`Ratio`（每垛箱数）时，触发满垛逻辑
   - 将`TagNo`保存到待关联Map中（无箱码模式）

**数据库操作要点**:
- 检查重复码需要查询`SmallSerialNumber`和`BigSerialNumber`两个字段
- 插入记录时设置所有必需字段的默认值
- 使用内存缓存管理`TagNo`和采集数量

#### 3.2.2 无箱码采集
**业务逻辑**:
1. 接收箱码采集设备数据（应为"null"）
2. 如果是无箱码模式且箱码不为"null"，提示错误并忽略
3. 系统生成唯一箱码：
   - 格式：订单号 + 时间戳 + 随机数
4. 其他逻辑与有箱码采集相同
5. 满垛后保存`TagNo`到`pendingNoBoxTagNoMap`，等待托盘码关联

**数据库操作要点**:
- 生成唯一箱码确保不重复
- 插入记录时设置`Type=2`（无箱码）
- 其他字段设置与有箱码采集相同

#### 3.2.3 有箱码托盘码关联
**业务逻辑**:
1. 接收托盘码（`BigSerialNumber`）和触发箱码（`SmallSerialNumber`）
2. 根据触发箱码查找`TagNo`：
   - 查询`CodeRelationUpload`表，找到对应`SmallSerialNumber`的记录
   - 获取该记录的`TagNo`
3. 验证箱数匹配：
   - 查询该`TagNo`下的实际采集箱数
   - 与前端传递的`Qty`（应采箱数）比较
   - 如果不匹配且未启用强制满垛，返回错误
   - 如果启用强制满垛（`forcePalletFlag`），忽略箱数差异
4. 查询该`TagNo`的最新`Qty`条记录（按`AddTime`降序）
5. 批量更新记录：
   - 设置`BigSerialNumber`（托盘码）
   - 生成虚拟垛标（`VirtualSerialNumber`）
   - 设置`IsVirtual=1`
   - 设置`IsUpload=0`（待上传）
   - **不更新`Status`字段**（保持原值）
6. 更新`Qty`字段：
   - 重新统计该`TagNo`下`BigSerialNumber`有值的数量
   - 更新所有记录的`Qty`字段
7. 更新统计字段：
   - 更新`ProductionOrder.OrderSumCount`（累加已生成虚拟垛标的数量）
   - 更新`ProductionOrderDetail.OrderCount`（累加已生成虚拟垛标的数量）
8. 清除缓存：
   - 清除`currentTagNoMap`中的`TagNo`（如果与当前采集的`TagNo`一致）
   - 清除`palletCountMap`中的计数
   - 清除`forcePalletFlagMap`中的强制满垛标记

**数据库操作要点**:
- 使用分页查询获取最新`Qty`条记录
- 批量更新使用IN条件更新多条记录
- 虚拟垛标生成规则：`#托盘码#_日期+随机数+C`
- 需要同时更新`ProductionOrder`和`ProductionOrderDetail`两个表的统计字段
- 码关联时不更新`Status`字段

#### 3.2.4 无箱码托盘码关联
**业务逻辑**:
1. 从`pendingNoBoxTagNoMap`获取待关联的`TagNo`
2. 验证箱数匹配（与有箱码逻辑相同）
3. 查询该`TagNo`下`BigSerialNumber`为空的最新`Qty`条记录
4. 批量更新记录（与有箱码逻辑相同）
5. 更新统计字段（与有箱码逻辑相同）
6. 清除缓存：
   - 清除`pendingNoBoxTagNoMap`中的`TagNo`
   - 清除`palletCountMap`中的计数
   - 清除`forcePalletFlagMap`中的强制满垛标记

**数据库操作要点**:
- 查询条件需要过滤`BigSerialNumber`为空或空字符串的记录
- 其他操作与有箱码托盘码关联相同

### 3.3 数据上传

#### 3.3.1 查询上传数据
**业务逻辑**:
1. 查询`ProductionOrderDetail`表获取订单列表
2. 对每个订单：
   - 查询`CodeRelationUpload`表中`BigSerialNumber`有值的记录数（完成数量）
   - 查询`ProductionOrder`表获取`OrderStatus`（生产状态）
3. 返回订单列表和完成数量

**数据库操作要点**:
- 查询订单列表时过滤`IsDel=0`的记录
- 完成数量统计需要查询`BigSerialNumber`不为空且不为空字符串的记录
- 关联查询`ProductionOrder`表获取订单状态

#### 3.3.2 上传数据到WMS
**业务逻辑**:
1. 查询`CodeRelationUpload`表中`IsUpload=0`（待上传）的记录
2. 调用WMS接口上传数据
3. 上传成功后：
   - 设置`IsUpload=1`（已上传）
   - 设置`UploadTime`（上传时间）
4. 上传失败：
   - 增加`ErrCount`（失败次数）
   - 设置`Msg`（错误信息）

**数据库操作要点**:
- 查询待上传记录时过滤`IsUpload=0`和`IsDel=0`的记录
- 上传成功后批量更新`IsUpload`和`UploadTime`字段
- 上传失败时更新`ErrCount`和`Msg`字段

### 3.4 码查询

#### 3.4.1 查询码关联关系
**业务逻辑**:
1. 根据输入的码（箱码、托盘码、虚拟垛标）查询`CodeRelationUpload`表
2. 如果输入的是箱码，查询`SmallSerialNumber`字段
3. 如果输入的是托盘码，查询`BigSerialNumber`字段
4. 如果输入的是虚拟垛标，查询`VirtualSerialNumber`字段
5. 返回关联的所有记录（同一`VirtualSerialNumber`下的所有记录）
6. 关联查询`ProductionOrderDetail`表获取产品信息

**数据库操作要点**:
- 根据不同的码类型查询不同的字段
- 如果查询到虚拟垛标，需要查询该虚拟垛标下的所有关联记录
- 关联查询`ProductionOrderDetail`表获取产品详细信息
- 按`AddTime`升序排列，保持时间顺序

### 3.5 操作日志

#### 3.5.1 记录操作日志
**业务逻辑**:
1. 记录操作人、操作模块、操作类型、操作对象、操作内容等
2. 记录操作前数据和操作后数据（JSON格式）
3. 记录操作结果（成功/失败）
4. 记录操作时间和设备信息

**数据库操作要点**:
- 插入`OperateLog`表记录
- 操作前后数据以JSON格式存储
- 记录操作时间精确到毫秒

## 4. 核心数据结构

### 4.1 内存缓存
- **currentTagNoMap**: 当前订单的TagNo缓存（key=订单号, value=TagNo）
- **palletCountMap**: 当前TagNo的采集数量缓存（key=TagNo, value=采集数量）
- **pendingNoBoxTagNoMap**: 无箱码模式待关联的TagNo缓存（key=订单号, value=TagNo）
- **forcePalletFlagMap**: 强制满垛标记缓存（key=订单号, value=是否强制满垛）

### 4.2 虚拟垛标生成规则
- 格式：`#托盘码#_日期+随机数+C`
- 日期格式：yyyyMMdd（8位数字）
- 随机数：3位数字（000-999）
- 示例：`#P20241201001#_20241201023C`

## 5. 接口设计

### 5.1 任务管理接口
- `POST /api/task/add` - 添加任务
- `PUT /api/task/update/{id}` - 更新任务
- `DELETE /api/task/delete/{id}` - 删除任务
- `POST /api/task/page` - 分页查询任务
- `POST /api/task/page/selectable` - 分页查询可选择的任务（排除已完成）
- `GET /api/task/get/{orderNo}` - 根据订单号获取任务
- `PUT /api/task/updateStatus/{id}/{status}` - 更新任务状态

### 5.2 码采集接口
- `POST /api/code/collect` - 箱码采集
- `POST /api/code/associate-pallet` - 托盘码关联（有箱码）
- `POST /api/code/collect-no-box` - 无箱码托盘码关联
- `GET /api/code/current-pallet-info/{taskId}` - 获取当前垛信息
- `GET /api/code/collected-count/{orderNo}` - 获取已采集箱数
- `GET /api/code/produced-pallet-count/{orderNo}` - 获取已生产垛数

### 5.3 数据上传接口
- `GET /api/data-upload/orders` - 查询上传订单列表
- `POST /api/data-upload/upload/{orderNo}` - 上传订单数据

### 5.4 码查询接口
- `POST /api/code/query` - 查询码关联关系

## 6. 事务管理

### 6.1 事务注解
所有涉及数据库写操作的方法都使用`@Transactional(rollbackFor = Exception.class)`注解，确保数据一致性。

### 6.2 事务范围
- 任务添加、更新、删除
- 码采集和关联
- 数据上传

## 7. 异常处理

### 7.1 业务异常
- 使用`ApiResult`统一返回格式
- 成功：返回200状态码和数据
- 失败：返回错误状态码和错误信息

### 7.2 异常捕获
- 所有业务方法使用try-catch捕获异常
- 记录异常日志
- 返回友好的错误信息

## 8. 数据库操作规范

### 8.1 查询规范
- 所有查询必须过滤`IsDel=0`的记录
- 使用MyBatis-Plus的`LambdaQueryWrapper`构建查询条件
- 分页查询使用`Page`对象
- 关联查询时注意性能优化

### 8.2 更新规范
- 逻辑删除使用`UpdateWrapper`设置`IsDel=1`
- 批量更新使用`LambdaUpdateWrapper`
- 更新操作需要记录操作日志
- 更新前需要验证数据是否存在

### 8.3 插入规范
- 设置默认值：`IsDel=0`，`Status=0`，`IsUpload=1`
- 设置时间字段：`AddTime=当前时间`
- 字符串字段如果为空，设置为空字符串（不是null）
- 所有必需字段必须设置值

## 9. 注意事项

### 9.1 并发控制
- 使用`ConcurrentHashMap`管理内存缓存
- 码采集和关联操作需要考虑并发安全
- 数据库操作使用事务保证一致性

### 9.2 数据一致性
- 码关联后需要同步更新多个表的统计字段
- 使用事务确保数据一致性
- 更新操作需要验证数据完整性

### 9.3 性能优化
- 使用分页查询避免一次性加载大量数据
- 使用索引优化查询性能
- 合理使用缓存减少数据库查询
- 批量操作时使用批量更新

### 9.4 状态管理
- 任务状态：0-待生产，1-生产中，2-已完成，5-提前结单
- 码状态：0-未完成，1-已完成，2-已解除（码关联时不更新此字段）
- 上传状态：0-同步，1-不同步
