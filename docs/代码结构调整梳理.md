# 代码结构调整梳理

> 依据：赋码采集关联系统T1.2.0（石湾版）- AI文档 第 5.5 节「版本与代码管理」

## 一、文档要求摘要

| 维度 | 建议 |
| -- | -- |
| **代码组织** | 按产线划分代码模块或文件夹（如 `zhimeizhai/`、`shiwan/`），共用部分抽离为公共模块 |
| **方案文档** | 每产线独立方案文档，版本号与迭代对应；文档变更可追溯 |
| **共用与差异** | 底层框架、UI 组件、业务工具类共用；产线特有逻辑（关联层级、包装比例、硬件配置）独立实现 |

## 二、当前项目结构现状

### 2.1 模块划分（按 DDD 分层，未按产线划分）

```
miduo-basebackend/
├── miduo-common          # 公共工具、DTO
├── miduo-entity          # 实体、枚举、请求/响应对象
├── miduo-domain          # 领域模型、领域服务
├── miduo-application    # 应用服务
├── miduo-infrastructure  # 持久化、Mapper、Repository 实现
├── miduo-controller      # REST API（后端）
├── miduo-frontend        # JavaFX 前端
└── miduo-bootstrap       # 启动入口
```

### 2.2 包结构（无产线维度）

```
com.miduo.cloud
├── common.*
├── entity.*
├── domain.code.*
├── domain.task.*
├── domain.device.*
├── application.code.*
├── application.task.*
├── infrastructure.*
├── controller.*
└── frontend.controller.*
```

### 2.3 业务模型现状（致美斋二级关联）

| 概念 | 当前实现 | 说明 |
|------|----------|------|
| 关联层级 | 托盘-箱（二级） | `CodeRelation`：tagNo(托盘)、smallSerialNumber(箱码)、ratio(拖箱比例) |
| 包装比例 | `ratio`（拖箱比例） | Task.ratio、TaskAddRequest.ratio |
| 任务类型 | type=1 有箱码 / type=2 无箱码 | 致美斋模式 |
| 软件形态 | 单软件 | MainWindow 集成任务、码替换、数据查询、数据上传、系统设置等 |

### 2.4 与文档要求的差距

| 文档要求 | 当前状态 | 差距 |
|----------|----------|------|
| 按产线划分（zhimeizhai/、shiwan/） | 无产线维度 | 代码未按产线组织 |
| 共用部分抽离 | 全部混在一起 | 共用与产线特有逻辑未分离 |
| 产线特有逻辑独立 | 仅实现致美斋逻辑 | 石湾四级关联、双软件、G/N/M 等未独立实现 |

## 三、产线差异对照

| 维度 | 致美斋产线 | 石湾产线 |
|------|------------|----------|
| 关联层级 | 二级：托盘-箱（1:N） | 四级：瓶-盒-箱-垛（1:M:N:G） |
| 包装比例 | ratio（拖箱比例） | G/N/M 可配置（例：G=6、N=4、M=70） |
| 软件形态 | 单软件 | 双软件：瓶盒关联(1号机)、盒箱垛关联(2号机) |
| 硬件配置 | 通用 | 最多 5 台相机、龙门架剔除、Modbus 等 |
| 特有功能 | 基础采集、码替换、数据上传 | 码包管理、码包比对、自动上传、指示灯控制、手工采集、取消关联等 |

## 四、建议的代码结构调整方案（模块级产线划分）

在 Maven 模块层面引入产线维度，共用部分保持独立模块。

```
miduo-basebackend/
├── miduo-common              # 共用：工具类、通用 DTO、常量
├── miduo-entity              # 共用：通用实体、枚举（可考虑拆出产线特有 entity）
├── miduo-infrastructure      # 共用：DB 访问、Mapper、设备驱动基础
│
├── miduo-productline-common  # 新增：产品线共用
│   ├── 底层框架、Spring/JavaFX 配置
│   ├── 通用 UI 组件（表格、表单、按钮、弹窗骨架）
│   ├── 操作帮助(P03-01)、系统设置(P03-02) 共用弹窗
│   └── 激活码、许可证等通用能力
│
├── miduo-zhimeizhai          # 致美斋产线模块
│   ├── domain/               # 二级关联领域模型
│   ├── application/          # 致美斋应用服务
│   ├── frontend/             # 致美斋主界面、控制器
│   └── 打包：致美斋单软件安装包
│
├── miduo-shiwan              # 石湾产线模块
│   ├── domain/               # 四级关联领域模型（瓶-盒-箱-垛）
│   ├── application/          # 石湾应用服务（码包、比对、双机协同等）
│   ├── frontend-bottle-box/  # 瓶盒关联(1号机) 界面
│   ├── frontend-box-pallet/  # 盒箱垛关联(2号机) 界面
│   └── 打包：两个安装包（1号机、2号机）
│
├── miduo-controller          # 共用或按需拆分
├── miduo-bootstrap           # 启动模块，按产线选择启动
```

**优点**：产线边界清晰，打包、依赖管理简单。  
**缺点**：模块数量增加，需梳理依赖关系。

## 六、需独立实现的产线特有逻辑

### 6.1 致美斋产线

| 项目 | 说明 |
|------|------|
| 关联层级 | 托盘-箱（1:N） |
| 包装比例 | ratio（拖箱比例） |
| 数据模型 | CodeRelation（tagNo、smallSerialNumber、ratio） |
| 主界面 | 单软件，任务+采集+码替换+数据查询+数据上传 |

### 6.2 石湾产线

| 项目 | 说明 |
|------|------|
| 关联层级 | 瓶-盒-箱-垛（1:M:N:G） |
| 包装比例 | G/N/M 可配置，不可写死 |
| 数据模型 | 需扩展/新增：瓶码、盒码、箱码、垛码及四级关联 |
| 软件形态 | 双软件：瓶盒关联(1号机)、盒箱垛关联(2号机) |
| 1号机 | P01-01 数据采集、P03-02 系统设置，访问 2 号机 DB |
| 2号机 | P02-01~08 共 8 个 Tab、P03-02 系统设置，本机 DB |
| 特有功能 | 码包管理、码包比对、自动上传、指示灯、手工采集、取消关联、剔除记录等 |
| 硬件 | 最多 5 台相机、龙门架剔除、Modbus |

## 七、调整步骤建议（不执行，仅规划）

1. **梳理共用与差异**  
   - 列出各模块中与产线无关的代码  
   - 列出致美斋、石湾各自特有逻辑及数据模型  

2. **抽离共用部分**  
   - 新建 `miduo-productline-common` 模块  
   - 迁移通用 UI、工具、弹窗、激活码等  

3. **引入产线维度**  
   - 创建 `zhimeizhai`、`shiwan` 模块  
   - 将现有致美斋逻辑归入 `zhimeizhai`  

4. **实现石湾产线**  
   - 在 `shiwan` 下实现四级关联、双软件、G/N/M、码包等  
   - 复用共用模块和弹窗  

5. **打包与启动**  
   - 致美斋：单安装包  
   - 石湾：瓶盒关联(1号机)、盒箱垛关联(2号机) 两个安装包

## 九、总结

- **现状**：项目按 DDD 分层组织，未按产线划分，仅实现致美斋二级关联。  
- **目标**：按产线划分代码，共用部分抽离，产线特有逻辑独立实现。  
- **方案**：采用**模块级产线划分**，便于后续多产线扩展和打包管理。  
- **实施**：先完成共用抽离与产线维度引入，再在石湾模块中实现四级关联与双软件。
