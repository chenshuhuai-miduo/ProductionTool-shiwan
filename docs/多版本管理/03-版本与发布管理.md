# 版本与发布管理

> 多软件版本下的代码版本、分支、构建与发布规范  
> 配合阅读：[多版本项目管理方案](./多版本项目管理方案.md)

---

## 1. 两种「版本」概念

| 概念 | 含义 | 示例 |
|------|------|------|
| **软件版本（Variant）** | 面向客户/产线的产品线，是可部署产品的「型号」 | 产线A、客户B、工厂01 |
| **发布版本（Release）** | 某软件版本的某次发布，有版本号、构建产物 | 产线A-v1.2.0、客户B-v2.0.0 |

- 一个 **软件版本** 会有多次 **发布**（1.0.0、1.1.0、1.2.0 …）。  
- 代码版本（Git 提交/标签）用来追溯：**某次发布** 对应 **哪份代码**。

---

## 2. 软件版本 ID 规范

### 2.1 命名

- **唯一**：全局唯一，如 `line-a`、`client-b`、`plant-01`。  
- **稳定**：一旦对外使用，不随意改名，避免配置、脚本、文档大面积修改。  
- **建议**：小写、数字+字母、可含 `-`，如 `line-a`、`wms-client-01`。

### 2.2 与配置、构建的对应

- 每个软件版本 ID 对应：  
  - 一份（或一组）版本配置：`config/variants/<软件版本ID>.yml`  
  - 一个可选变体模块：`miduo-variant-<软件版本ID>`  
  - 构建时通过「软件版本 ID」选择配置和模块，产出该版本的包。

---

## 3. 发布版本号规范

### 3.1 语义化版本（推荐）

- 格式：`主版本.次版本.修订号`，如 `1.2.0`。  
  - **主版本**：不兼容的变更（如协议、重大架构）。  
  - **次版本**：向后兼容的功能新增。  
  - **修订号**：向后兼容的问题修复、小优化。  
- 与软件版本 ID 组合表示一次发布：`产线A-v1.2.0`、`client-b-v2.0.0`。

### 3.2 预发布与构建号（可选）

- 预发布：`1.2.0-alpha.1`、`1.2.0-beta.2`。  
- 构建号：`1.2.0+build.20260126`，用于同一语义版本多次构建的区分。

## 4. Git 分支策略

### 4.1 主分支

| 分支 | 用途 | 合并方向 |
|------|------|----------|
| `main` | 生产就绪，仅用于发布与打标签 | 从 `develop` 或 `release/*` 合并 |
| `develop` | 日常开发与集成，通用功能、扩展开发 | 从 `feature/*` 合并；向 `release/*` 或 `main` 合并 |

### 4.2 功能分支

- 分支名：`feature/<简短描述>` 或 `feature/<JIRA-ID>-<描述>`。  
- 基于 `develop` 创建，完成后合并回 `develop`。  
- 功能应尽量落在「核心」或「扩展」，避免版本专属逻辑进核心。

### 4.3 发布/版本分支（按需）

- 当 **某一软件版本** 有较长时间的专属迭代（如大定制、验收前冻结），可开：  
  - `release/<软件版本ID>` 或 `release/<软件版本ID>-v<版本号>`  
- 用途：  
  - 该版本的缺陷修复、小改动；  
  - 从 `develop` 定期合并，同步通用代码。  
- 发布完成后，将发布分支合并回 `main`，并打标签；视情况合并回 `develop`。  
- **注意**：不要为 25 个版本长期维护 25 条分支；优先用「配置 + 变体」区分版本，仅对迭代多的少数版本使用发布分支。

### 4.4 热修复（Hotfix）

- 分支名：`hotfix/<描述>` 或 `hotfix/<版本号>`。  
- 基于 `main` 创建，修复后合并回 `main` 和 `develop`，并打新标签。

### 4.5 分支与「软件版本」的关系

- **软件版本** 与「分支」无固定绑定。  
- 区分版本主要靠 **配置 + 变体模块**；  
- **发布分支** 仅用于「某个软件版本」的发布节奏与冻结，其内容仍通过「该版本的配置」决定包含哪些扩展/变体。

---

## 5. 标签（Tag）规范

### 5.1 发布标签

- 格式建议：`<软件版本ID>/v<发布版本号>` 或 `<软件版本ID>-v<发布版本号>`。  
- 示例：`line-a/v1.2.0`、`client-b-v2.0.0`。  
- 每次 **可发布** 的构建打一个标签，便于：  
  - 追溯该发布对应的代码版本  
  - 回滚、重新构建

### 5.2 核心/通用基线标签（可选）

- 若希望标记「某次重大通用能力发布」，可打：`core/v1.0.0` 等，用于各软件版本在发布分支上对齐基线。

## 6. 构建与制品

### 6.1 构建输入

- **软件版本 ID**（必选）：决定用哪份 `config/variants/<id>.yml`、引入哪些扩展与变体。  
- **发布版本号**（推荐）：写入包元数据、文件名。  
- **Git 引用**：标签或 commit，用于追溯。

### 6.2 构建过程（概念）

1. 读取 `config/variants/<软件版本ID>.yml`（及 `variant-common.yml`）。  
2. 根据配置激活 Maven profile 或选择模块集合（核心 + 配置中列出的 extension + 对应 variant）。  
3. 执行 Maven 构建，产出该版本的 jar/安装包。  
4. 制品命名建议：`赋码采集关联系统-<软件版本ID>-<发布版本号>.jar` 或带日期的形式。

### 6.3 制品元数据（推荐）

- 在 jar manifest 或单独文件中记录：  
  - 软件版本 ID  
  - 发布版本号  
  - Git 标签或 commit  
  - 构建时间  
- 便于部署、运维和问题排查时追溯。

---

## 7. 发布流程（建议）

### 7.1 常规发布（从 develop）

1. 在 `develop` 上完成功能开发与测试。  
2. 需要发布时，从 `develop` 拉取或使用指定 commit，用 **目标软件版本 ID** 执行构建。  
3. 对构建产物做该版本的发布测试。  
4. 通过后打标签：`<软件版本ID>/v<发布版本号>`。  
5. 将用于发布的 commit 合并到 `main`（若未在 main 上直接打标签）。

### 7.2 带发布分支的发布（某版本有专属迭代）

1. 从 `develop` 拉取 `release/<软件版本ID>`，在该分支上做该版本的专属开发与修复。  
2. 定期将 `develop` 合并到该发布分支，同步通用代码。  
3. 构建时使用 **该软件版本 ID** 的配置，从发布分支构建。  
4. 测试通过后打标签，将发布分支合并到 `main`，可选合并回 `develop`。

### 7.3 多版本同时发布

- 同一批通用改动（同一 Git 提交或标签）可被多个软件版本使用。  
- 对各软件版本分别：用其 **软件版本 ID** 构建 → 测试 → 打各自标签（如 `line-a/v1.2.0`、`client-b/v2.0.0`）。  
- 这样既共享代码，又保持各版本发布版本号独立。

## 8. 版本与发布信息记录

### 8.1 发布记录表（建议）

在 `docs` 或项目 Wiki 维护表格，例如：

| 软件版本 ID | 发布版本号 | Git 标签 / Commit | 构建日期 | 备注 |
|-------------|------------|-------------------|----------|------|
| line-a      | 1.2.0      | line-a/v1.2.0     | 2026-01-26 | 新增报表 |
| client-b    | 2.0.0      | client-b-v2.0.0   | 2026-01-25 | 大版本升级 |

### 8.2 软件版本清单

- 维护「软件版本 ID ↔ 说明、使用方、配置路径」的清单，便于新人与交付管理。  
- 可与 `config/variants/` 下的文件名保持一致。

---

## 9. 与「多版本项目管理」的对应关系

- **代码版本**：由 Git 分支与标签管理，保证「某次发布 = 某份代码」可追溯。  
- **软件版本**：由「软件版本 ID + 配置 + 变体」决定，不依赖长期分支。  
- **发布版本**：软件版本 ID + 发布版本号 + 标签，对应唯一构建产物与部署实例。  
- **通用代码复用**：通用逻辑在核心/扩展中维护，各版本通过「同一仓库 + 不同配置与变体」复用，无需为每个软件版本单独建库或长期分支。
