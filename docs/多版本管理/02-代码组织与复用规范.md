# 代码组织与复用规范

> 多版本场景下，如何组织代码、如何保证通用代码便于复用  
> 配合阅读：[多版本项目管理方案](./多版本项目管理方案.md)

---

## 1. 三层结构

### 1.1 核心层（Core）

- **定义**：所有软件版本都必须具备的能力，无例外。
- **位置**：现有 `miduo-common`、`miduo-entity`、`miduo-domain`、`miduo-infrastructure`、`miduo-application`、`miduo-controller`、`miduo-frontend`、`miduo-bootstrap` 等。
- **规则**：
  - 不依赖任何「扩展层」「变体层」模块。
  - 不包含「仅个别版本需要」的逻辑；若有，应抽到扩展层或变体层。
  - 对外提供接口（API、SPI），由扩展/变体实现或扩展，而不是在核心写死分支（如 `if (variant == A)`）。

**复用方式**：所有版本构建时都包含核心模块，天然复用。

---

### 1.2 扩展层（Optional / Extension）

- **定义**：可选能力，部分软件版本需要，部分不需要。
- **位置**：新建 `miduo-extension-<能力名>`，例如：
  - `miduo-extension-report`：高级报表
  - `miduo-extension-protocol-wms`：某 WMS 上传协议
  - `miduo-extension-device-xxx`：某类设备驱动
- **规则**：
  - 只依赖核心层（及同一层其他扩展，若必要且无循环）。
  - 通过接口/SPI 与核心交互，核心通过配置或 SPI 加载扩展，而不是核心直接依赖扩展模块。
  - 单个扩展模块尽量内聚，避免一个扩展依赖大量其他扩展。

**复用方式**：在「版本配置」中声明该版本启用哪些扩展模块，构建时只引入声明的扩展，实现「写一次，多版本可选复用」。

---

### 1.3 变体层（Variant）

- **定义**：仅某一（或极少数）软件版本独有的实现、资源、配置。
- **位置**：新建 `miduo-variant-<软件版本ID>`，例如 `miduo-variant-line-a`。
- **规则**：
  - 可依赖核心层和本版本启用的扩展层。
  - 仅包含「该版本特有」的内容；若多个版本共用，应提升到扩展层或核心层。
  - 尽量薄：优先用配置 + 扩展层满足需求，变体只做「无法抽象成扩展」的差异。

**复用方式**：每个软件版本对应一个变体模块（或共享变体），构建时只包含该版本的变体，避免无关代码进入包内。

---

## 2. 依赖方向与复用关系

```
                     ┌─────────────────────────────────────┐
                     │            核心层 (Core)              │
                     │  不依赖扩展/变体；定义接口与扩展点      │
                     └─────────────────────────────────────┘
                                          ▲
                      ┌───────────────────┼───────────────────┐
                      │                   │                   │
                      │ 只依赖核心         │ 只依赖核心         │
                      ▼                   ▼                   ▼
         ┌────────────────────┐ ┌────────────────────┐ ┌────────────────────┐
         │ 扩展层 (Extension)  │ │ 扩展层 (Extension)  │ │ 扩展层 (Extension)  │
         │ 可选能力 A         │ │ 可选能力 B         │ │ 可选能力 C         │
         └────────────────────┘ └────────────────────┘ └────────────────────┘
                      ▲                   ▲                   ▲
                      │                   │                   │
                      └───────────────────┼───────────────────┘
                                          │ 依赖核心 + 本版本启用的扩展
                                          ▼
                     ┌─────────────────────────────────────┐
                     │ 变体层 (Variant)                     │
                     │ 版本专属实现与资源，尽量薄           │
                     └─────────────────────────────────────┘
```

- **核心**：被所有版本复用，禁止依赖扩展/变体。  
- **扩展**：被「需要该能力的版本」复用，不依赖变体。  
- **变体**：仅对应版本使用，可依赖核心和若干扩展。

---

## 3. 如何把「通用代码」放到可复用位置

### 3.1 判断归属

| 现象 | 建议归属 |
|------|----------|
| 所有版本都要用 | 核心层 |
| 多个版本要用，但不是全部 | 扩展层（一个或多个 extension 模块） |
| 仅一个或极少数版本要用 | 变体层 |

### 3.2 从「复制粘贴」到「复用」

- 若某逻辑已在「变体 A」中存在，现「变体 B」也要用：  
  - 将逻辑抽到 **核心层**（若所有版本都需要）或 **扩展层**（若仅部分版本需要）。  
  - 变体 A/B 改为依赖核心或该扩展，删除重复实现。  
- 若某逻辑在多个变体里略有差异：  
  - 在核心或扩展中定义 **接口 + 少量抽象**，各变体提供不同实现，通过配置选择实现类。

### 3.3 接口与扩展点

- 核心层定义 **接口**（如 `UploadProtocol`、`DeviceDriver`），实现可在核心（默认实现）或扩展、变体中。  
- 通过 **配置**（类名、Spring 配置）或 **Java SPI** 在运行时选择实现，避免核心层 `import` 扩展/变体模块。  
- 这样新增「可选能力」时只需新增扩展模块并实现接口，无需改核心代码，便于在多版本间复用。

---

## 4. 命名与包结构

### 4.1 模块命名

- 核心层：保持现有 `miduo-*`。  
- 扩展层：`miduo-extension-<能力简称>`，如 `miduo-extension-report`。  
- 变体层：`miduo-variant-<软件版本ID>`，如 `miduo-variant-line-a`。

### 4.2 包名

- 核心：`com.miduo.cloud.*`（保持不变）。  
- 扩展：`com.miduo.cloud.extension.<能力>.*`，如 `com.miduo.cloud.extension.report.*`。  
- 变体：`com.miduo.cloud.variant.<版本ID>.*`，如 `com.miduo.cloud.variant.linea.*`。

### 4.3 避免「版本」出现在核心包名中

- 核心层包名不应包含 `linea`、`clientb` 等具体版本 ID，避免被误认为「仅某版本使用」。  
- 版本相关分支逻辑尽量收口到扩展/变体或配置，而不是在核心包下按版本建子包。

---

## 5. 配置与功能开关

### 5.1 版本级配置

- 各软件版本使用独立配置文件（如 `config/variants/<id>.yml`），不把「版本 ID」写死在代码里。  
- 构建时通过 **软件版本 ID** 选择使用哪份配置，从而决定加载哪些扩展、哪些变体。

### 5.2 功能开关（可选）

- 若采用功能矩阵：  
  - 在配置或数据库中维护「功能 ID ↔ 是否启用」。  
  - 核心/扩展代码根据「功能是否启用」决定是否注册某功能、是否显示某入口，而不是根据「版本 ID」硬编码。

---

## 6. 测试与复用安全

### 6.1 核心层

- 核心层变更必须有 **单元测试 / 集成测试** 覆盖。  
- 若有「核心 + 若干扩展」的经典组合，可建 **组合用例**，保证通用代码修改后这些组合仍正常。

### 6.2 扩展层

- 每个扩展模块应有自身测试；若被多版本复用，建议在 CI 中至少对一个使用该扩展的版本做构建和冒烟测试。

### 6.3 变体层

- 变体模块的测试可侧重「该版本专属逻辑」；集成测试时使用对应版本配置做一次完整构建与启动验证。

---

## 7. 文档与清单

### 7.1 功能归属清单

- 建议在 `docs` 或 `config/features` 中维护一份 **功能/模块清单**：  
  - 功能名称  
  - 所属层（核心/扩展/变体）  
  - 所属模块  
  - 使用该功能的软件版本 ID 列表  
- 便于：新版本选型、重构时判断逻辑应放在哪一层、排查「某版本有没有某功能」。

### 7.2 复用记录

- 当把一段「原属某变体」的代码提升到扩展或核心时，在提交说明或文档中简要记录：  
  - 原归属版本  
  - 新归属模块  
  - 当前复用该能力的版本列表  

---

## 8. 小结

| 目标 | 做法 |
|------|------|
| 通用代码复用 | 通用逻辑放在核心层或扩展层；核心不依赖扩展/变体 |
| 多版本可选能力 | 扩展层 + 版本配置决定「哪些扩展参与构建」 |
| 版本专属差异 | 变体层 + 配置；变体尽量薄，能抽就抽到扩展/核心 |
| 安全复用 | 接口与实现分离、配置驱动、测试覆盖核心与典型组合 |
| 可维护性 | 统一命名、功能归属清单、依赖方向约束 |
