# 产线采集软件 - 多版本项目管理方案

> 适用于 25+ 软件版本、通用代码复用、各版本独立迭代的场景  
> 最后更新：2026-01-26

---

## 1. 目标与约束

### 1.1 目标

- **通用代码复用**：核心能力集中维护，一次修改多版本受益
- **版本独立迭代**：每个软件版本可独立需求、独立发布，互不阻塞
- **可追溯**：任一部署版本可追溯到代码版本、功能清单、构建产物
- **可扩展**：新增软件版本成本低，不破坏现有版本

### 1.2 约束

- 预计至少 **25 个软件版本** 用于部署
- 各版本包含 **通用功能** 与 **特定版本功能**
- 通用代码需在多个软件版本中 **安全、方便地复用**
- 每个软件版本有 **自己的迭代需求**，发布节奏可能不同

### 1.3 术语

| 术语 | 含义 |
|------|------|
| **软件版本（Variant）** | 面向某一客户/产线/场景的可部署产品，如「产线A-v1」「客户B-v2」 |
| **通用功能** | 多个软件版本共用的能力，放在核心/公共模块 |
| **特定版本功能** | 仅某一（或少数）软件版本需要的能力，放在扩展/变体模块 |
| **代码版本** | Git 提交、分支、标签等，用于追溯和发布 |
| **发布版本** | 某软件版本的某次发布，如「产线A-v1.2.0」 |

## 2. 总体策略

### 2.1 策略概览

采用 **「核心 + 变体」** 与 **「单仓库 + 版本配置」** 结合：

1. **单仓库（Monorepo）**  
   所有软件版本共用同一代码库，便于复用和统一升级。

2. **分层代码**  
   - **核心层**：通用业务与技术能力，所有版本依赖  
   - **扩展层**：可选功能模块，按需被各版本引入  
   - **变体层**：仅某软件版本使用的实现与配置  

3. **版本通过「配置 + 构建」区分**  
   用「软件版本 ID + 构建配置」决定：  
   依赖哪些模块、启用哪些功能、打哪个包，而不是为每个版本单独建仓库或长期分支。

4. **发布与追溯**  
   每个可部署包对应：软件版本 ID + 版本号 + Git 标签（或 commit），便于追溯和回滚。

## 3. 代码组织

### 3.1 目录与模块划分（推荐）

在现有 Maven 多模块基础上，增加「扩展」与「变体」概念，不改变现有分层（controller/application/domain 等）：

```
productiontools/
├── pom.xml
├── docs/                          # 文档（含本方案）
├── config/                        # 版本级配置（与代码分离）
│   ├── variants/                  # 各软件版本配置
│   │   ├── variant-common.yml     # 全版本默认
│   │   ├── line-a.yml
│   │   ├── line-b.yml
│   │   └── ...
│   └── features/                  # 功能特性定义（可选）
│       └── feature-matrix.yml
│
├── miduo-common/                  # 【核心】公共 DTO、工具
├── miduo-entity/                  # 【核心】实体、枚举（可再拆 core/optional）
├── miduo-domain/                  # 【核心】领域模型与服务
├── miduo-infrastructure/          # 【核心】基础设施实现
├── miduo-application/             # 【核心】应用服务
├── miduo-controller/              # 【核心】REST API
├── miduo-frontend/                # 【核心】JavaFX 主框架
├── miduo-bootstrap/               # 【核心】启动与通用配置
│
├── miduo-extension-*/             # 【扩展】可选功能模块（按需）
│   ├── miduo-extension-report/    # 示例：高级报表
│   ├── miduo-extension-protocol-x/ # 示例：某上传协议
│   └── ...
│
└── miduo-variant-*/               # 【变体】某软件版本专属
    ├── miduo-variant-line-a/      # 产线A 专属实现与资源
    ├── miduo-variant-line-b/
    └── ...
```

- **核心层**：保持现有模块，所有软件版本构建时都包含。  
- **扩展层**：`miduo-extension-*`，只在「需要该功能的软件版本」的构建中引入。  
- **变体层**：`miduo-variant-*`，仅对应某一软件版本（或少量版本）使用，用于该版本的定制逻辑、配置、资源。

### 3.2 通用代码复用原则

- **通用功能**  
  只放在 **核心层** 或 **扩展层**。  
  - 核心层：所有版本必选。  
  - 扩展层：通过「版本配置」声明是否引入，实现「一次开发，多版本可选复用」。

- **特定版本功能**  
  放在 **变体模块**（`miduo-variant-xxx`）。  
  - 若后续多个版本需要，再考虑抽到扩展层或核心层，避免复制粘贴。

- **依赖方向**  
  核心 ← 扩展 ← 变体。  
  核心不依赖扩展/变体；扩展不依赖变体；变体可依赖核心和部分扩展。

- **接口与实现分离**  
  核心层定义接口（如上传协议、设备驱动 SPI），扩展/变体提供实现，通过配置或 SPI 加载，便于替换和按版本选配。

详见：[代码组织与复用规范](./代码组织与复用规范.md)。

## 4. 版本与发布管理

### 4.1 软件版本标识

- **软件版本 ID**：唯一、稳定，用于配置、构建、发布，如 `line-a`、`client-b`、`plant-01`。  
- **发布版本号**：建议语义化，如 `1.2.0`（主.次.修订），与软件版本 ID 组合为「产线A-v1.2.0」。

### 4.2 代码版本（Git）策略

- **主分支**  
  - `main`：生产就绪，用于发布打标签。  
  - `develop`：日常开发与集成，通用功能与扩展在此开发。

- **版本分支（按需）**  
  - 某软件版本有大量专属需求时，可开长期分支，如 `release/line-a`。  
  - 定期将 `develop` 合并到该分支，保证通用代码更新能流入该版本。  
  - 避免为 25 个版本开 25 个长期分支；优先用「配置 + 变体模块」区分版本。

- **标签**  
  - 每次可发布构建打标签：`<variant-id>-v<version>` 或 `release/<variant-id>/<version>`，便于追溯。

### 4.3 构建与打包

- **输入**  
  - 软件版本 ID  
  - （可选）发布版本号、环境（dev/test/prod）

- **过程**  
  - 读取该软件版本的配置文件（如 `config/variants/line-a.yml`），得到：  
    - 启用的扩展模块列表  
    - 启用的变体模块  
    - 应用名、端口、资源等  
  - Maven 按配置激活 profile 或选择模块，执行构建，产出该版本的安装包/jar。

- **产出**  
  - 每个部署包带元数据：软件版本 ID、发布版本号、Git 标签（或 commit）、构建时间。

详见：[版本与发布管理](./版本与发布管理.md)。

## 5. 配置管理

### 5.1 版本配置文件

- **位置**：`config/variants/<软件版本ID>.yml`（或 .properties）。  
- **内容**：  
  - 模块开关（哪些 extension/variant 参与构建）  
  - 应用名、端口、日志名  
  - 功能开关（若使用特性矩阵）  
  - 默认配置项（数据库、设备等可由部署时覆盖）

- **通用默认**：`config/variants/variant-common.yml` 为所有版本共享默认值，各版本文件只写差异。

### 5.2 功能特性矩阵（可选）

- 用 `config/features/feature-matrix.yml` 或表格维护：  
  「功能项 ↔ 软件版本」对应关系。  
- 构建或运行时可根据该矩阵决定是否加载某扩展、是否显示某菜单等，便于产品与交付管理。

## 6. 实施节奏建议

### 6.1 第一阶段：奠定复用基础（当前）

- 保持现有核心模块结构，明确「核心 / 扩展 / 变体」边界。  
- 在 `docs` 中记录：哪些是通用功能、哪些是某版本特有。  
- 引入 `config/variants/`，先为 1～2 个软件版本建立配置与构建流程。

### 6.2 第二阶段：支持多版本构建

- 用 Maven profile 或多模块组合，根据 `config/variants/<id>.yml` 为不同软件版本打不同包。  
- 建立「软件版本 ID → 构建产物」的命名与归档规范。  
- 为每个可发布构建打 Git 标签。

### 6.3 第三阶段：扩展与变体规模化

- 将已重复的「版本专属逻辑」抽到扩展层或核心层。  
- 新增软件版本时，优先通过「复制并修改 variant 配置 + 少量变体模块」完成，避免新仓库。

### 6.4 第四阶段：自动化与治理

- CI/CD 按配置矩阵批量构建 25+ 版本，产出到制品库。  
- 定期审查：通用能力是否都在核心/扩展、变体是否只保留真正专属部分。

## 7. 风险与对策

| 风险 | 对策 |
|------|------|
| 通用代码被误改导致多版本故障 | 核心层变更需评审、自动化测试覆盖核心用例、关键版本在 CI 中构建 |
| 变体代码膨胀，难以复用 | 定期做「变体 → 扩展/核心」提升，在文档中维护「功能归属」 |
| 版本过多，构建慢 | 按版本分组构建、增量构建、只对变更影响的版本重跑 |
| 配置漂移，版本行为不一致 | 配置纳入仓库与 Code Review，用 schema 或模板校验 |
