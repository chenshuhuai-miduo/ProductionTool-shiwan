# 软件版本配置与构建说明

> 多版本场景下，如何为每个软件版本做配置、如何执行构建  
> 配合阅读：[多版本项目管理方案](./多版本项目管理方案.md)、[版本与发布管理](./版本与发布管理.md)

---

## 1. 版本配置目录

### 1.1 推荐结构

```
config/
├── variants/                    # 各软件版本配置
│   ├── variant-common.yml     # 全版本默认（端口、日志等）
│   ├── line-a.yml             # 产线A
│   ├── line-b.yml             # 产线B
│   ├── client-c.yml           # 客户C
│   └── ...                    # 共 25+ 软件版本
└── features/                   # 功能特性定义（可选）
    └── feature-matrix.yml     # 功能 ↔ 版本 对应表
```

### 1.2 通用默认配置（variant-common.yml）

所有软件版本共享的默认值，各版本文件只写「差异项」。

```yaml
# config/variants/variant-common.yml
# 全版本默认配置，各 variant 文件可覆盖

app:
  name: 赋码采集关联系统
  server:
    port: 8080

# 参与构建的模块（核心层始终包含，此处仅列出扩展与变体）
build:
  extensions: []          # 默认不启用扩展模块
  variant: null           # 默认无变体，或写通用变体 ID

# 功能开关（若使用功能矩阵）
features:
  taskManagement: true
  codeCollect: true
  dataUpload: true
  codeQuery: true
  codeReplace: true
  operateLog: true
  # 可选功能默认关闭，各版本按需开启
  advancedReport: false
  customUploadProtocol: false
```

### 1.3 单版本配置示例（line-a.yml）

仅写与 `variant-common.yml` 的差异。

```yaml
# config/variants/line-a.yml
# 产线A 软件版本

# 继承通用默认（概念上合并，实现时可用 YAML 锚点或程序合并）
# 以下为覆盖或新增

app:
  name: 赋码采集关联系统-产线A

build:
  extensions:
    - report              # 启用高级报表扩展
  variant: line-a          # 使用 miduo-variant-line-a

features:
  advancedReport: true     # 产线A 需要高级报表
```

## 2. 构建方式（概念）

### 2.1 输入

- **软件版本 ID**：必填，如 `line-a`。  
- **发布版本号**（可选）：如 `1.2.0`，用于制品名和元数据。  
- **Git 引用**（可选）：如 `HEAD`、`line-a/v1.2.0`，用于追溯。

### 2.2 过程（Maven 思路）

1. 读取 `config/variants/variant-common.yml` 与 `config/variants/<软件版本ID>.yml`，合并得到该版本的完整配置。  
2. 根据配置中的 `build.extensions`、`build.variant`：  
   - 激活对应 Maven profile，或  
   - 选择要包含的模块（核心 + 列出的 extension + 对应 variant）。  
3. 执行 `mvn clean package -P<profile>` 或等价命令，产出该版本的 jar/安装包。  
4. 制品命名示例：`赋码采集关联系统-<软件版本ID>-<发布版本号>.jar`。

### 2.3 命令行示例（伪代码）

```bash
# 构建「产线A」版本，发布版本号 1.2.0
./scripts/build-variant.sh line-a 1.2.0

# 或使用 Maven profile（需在 pom 中定义 profile 与模块组合）
mvn clean package -Pvariant-line-a -Drelease.version=1.2.0
```

## 3. Maven 多模块与 Profile 建议

### 3.1 核心模块（始终参与）

- 现有 `miduo-common`、`miduo-entity`、`miduo-domain`、`miduo-infrastructure`、`miduo-application`、`miduo-controller`、`miduo-frontend`、`miduo-bootstrap` 等，所有版本构建时都包含。

### 3.2 扩展/变体模块（按配置引入）

- 在父 `pom.xml` 中为每个 **扩展模块**、**变体模块** 定义 `<module>`，但可用 **profile** 控制是否参与构建：  
  - 例如 `-Pvariant-line-a` 时，将 `miduo-extension-report`、`miduo-variant-line-a` 加入构建，并让 `miduo-bootstrap` 依赖它们。  
- 或：  
  - 使用「按 profile 激活的 optional 依赖」：核心/启动模块通过 optional 依赖扩展与变体，构建时根据 profile 打开对应依赖。  

这样「通用代码」始终在核心/扩展里，通过配置和 profile 决定「每个软件版本」最终包含哪些模块。

## 4. 功能矩阵（可选）

### 4.1 用途

- 维护「功能项 ↔ 软件版本」对应关系，便于：  
  - 产品/交付确认某版本是否有某功能  
  - 构建或运行时根据矩阵决定是否加载某扩展、是否显示某菜单  

### 4.2 示例（feature-matrix.yml）

```yaml
# config/features/feature-matrix.yml
# 功能 ↔ 软件版本（仅示例，实际按需增删）

features:
  - id: taskManagement
    name: 任务管理
    layer: core
    variants: all              # 所有版本

  - id: codeCollect
    name: 码采集与关联
    layer: core
    variants: all

  - id: advancedReport
    name: 高级报表
    layer: extension
    module: miduo-extension-report
    variants:
      - line-a
      - client-b

  - id: customProtocolX
    name: 定制上传协议X
    layer: variant
    variants:
      - line-a
```

- `variants: all` 表示核心功能，所有版本都有。  
- 其他功能列出使用该功能的软件版本 ID，便于生成「某版本功能清单」和构建时选模块。

## 5. 软件版本清单（建议维护）

在 `docs` 或 `config/` 下维护一份「软件版本清单」，便于管理和新人上手。

| 软件版本 ID | 说明 | 配置文件 | 变体模块 | 备注 |
|-------------|------|----------|----------|------|
| line-a      | 产线A | config/variants/line-a.yml | miduo-variant-line-a | 含高级报表 |
| line-b      | 产线B | config/variants/line-b.yml | - | 仅核心 |
| client-c    | 客户C | config/variants/client-c.yml | miduo-variant-client-c | 定制协议 |
