# 复用代码与各产线独立开发 — 项目结构调整方案

> 依据：[01-多版本项目管理方案](d:\productiontools\docs\多版本管理\01-多版本项目管理方案.md)、[02-代码组织与复用规范](d:\productiontools\docs\多版本管理\02-代码组织与复用规范.md)、[04-软件版本配置与构建说明](d:\productiontools\docs\多版本管理\04-软件版本配置与构建说明.md)

## 一、目标与原则（与文档对齐）

| 目标 | 做法 |
|------|------|
| 通用代码复用 | 通用逻辑放在**核心层**；多版本可选能力放在**扩展层**；核心不依赖扩展/变体 |
| 各产线独立迭代 | 每条产线对应一个**变体模块** + 一份**版本配置**，构建时只包含该变体 |
| 可追溯 | 每个可部署包对应：软件版本 ID + 发布版本号 + Git 标签 |
| 可扩展 | 新产线 = 新增 `config/variants/<id>.yml` + 新增 `miduo-variant-<id>`，不改核心 |

**依赖方向**（02 文档）：核心 ← 扩展 ← 变体；核心不依赖扩展/变体。

## 二、总体目录与模块划分

在现有 8 个核心模块基础上，增加 **config/**、**扩展模块**、**变体模块**，核心模块职责收紧为「仅通用 + 接口」。

```
productiontools/
├── pom.xml
├── config/                              # 【新增】版本级配置，与代码分离
│   ├── variants/
│   │   ├── variant-common.yml           # 全版本默认
│   │   ├── zhimeizhai.yml               # 致美斋
│   │   ├── shiwan-m1.yml                # 石湾 1 号机
│   │   └── shiwan-m2.yml                # 石湾 2 号机
│   └── features/
│       └── feature-matrix.yml           # 功能 ↔ 版本 矩阵（可选）
│
├── miduo-common/                        # 【核心】公共 DTO、工具
├── miduo-entity/                        # 【核心】通用实体/枚举/请求响应形状
├── miduo-domain/                        # 【核心】领域接口与通用领域逻辑
├── miduo-infrastructure/                # 【核心】通用持久化接口 + 致美斋实现（见下）
├── miduo-application/                  # 【核心】通用应用服务 + SPI 接口
├── miduo-controller/                    # 【核心】通用 REST 接口
├── miduo-frontend/                      # 【核心】通用 JavaFX 框架与共用页面
├── miduo-bootstrap/                     # 【核心】启动入口，按 profile 加载变体
│
├── miduo-extension-*/                   # 【扩展】可选能力（按需新建）
│   └── （示例：miduo-extension-report）
│
└── miduo-variant-*/                     # 【变体】产线专属
    ├── miduo-variant-zhimeizhai/        # 致美斋
    └── miduo-variant-shiwan/            # 石湾（内含 m1/m2 或拆成两个 variant）
```

- **复用代码**：全部在 **核心层** 与 **扩展层**；各产线通过依赖核心 + 可选扩展获得复用。
- **各产线独立开发**：产线专属代码只在 **变体模块** 与 **config/variants/<id>.yml** 中，独立迭代、独立构建。

## 三、核心层：复用代码归属与包结构

核心层**不出现**产线 ID（如 zhimeizhai、shiwan），只保留「所有版本都要用」或「通过接口/SPI 扩展」的内容。

### 3.1 miduo-common（保持不动）

- **职责**：工具类、通用 DTO（ApiResult、PageInput、PageOutput）、IniFileUtil。
- **包名**：`com.miduo.cloud.common.*`
- **复用方式**：所有版本构建时均包含，无需改动。

### 3.2 miduo-entity（核心：通用形状 + 接口用 DTO）

- **职责**：所有产线共用的请求/响应形状、枚举、通用实体；**不**放产线专属表对应的 PO。
- **建议包结构**：

```
com.miduo.cloud.entity/
├── dto/                     # 通用 API 用 DTO（分页、操作结果等）
│   ├── ApiResult 等已在 common 的可在 common，此处仅实体相关
├── enums/                   # 通用枚举（OperateTypeEnum、ModuleNameEnum、License 等）
├── po/                      # 仅「所有版本共用表」的 PO，如 OperateLog
└── （致美斋专用 PO 如 CodeRelationPO、ProductionOrderPO 迁移到变体，见下）
```

- **归属原则**：若某 DTO/PO 仅致美斋或仅石湾使用，则移至对应变体模块的 entity 包。

### 3.3 miduo-domain（核心：接口与通用领域逻辑）

- **职责**：领域接口（如 CodeRepository、TaskRepository）、通用领域服务（格式校验、重码检测等可抽象部分）。
- **包名**：`com.miduo.cloud.domain.*`
- **规则**：核心只定义接口和通用算法；产线专属的「关联层级、包装比例」等由变体实现或通过 SPI 注入。

### 3.4 miduo-infrastructure（核心：接口 + 致美斋实现）

- **职责**：  
  - 核心：持久化接口、OperateLog 等**全版本共用表**的实现。  
  - 致美斋：当前 SQL Server 实现（CodeRelation、ProductionOrder、IoConfig 等）保留在核心，视为「默认实现」；或迁到 `miduo-variant-zhimeizhai`（见 3.4 两种方案）。
- **推荐**（与文档一致、减少核心体积）：  
  - 核心只保留：**接口** + **所有版本共用的持久化**（如 OperateLog）。  
  - 致美斋专属表（CodeRelationUpload、ProductionOrder 等）的 PO/Mapper 迁到 `miduo-variant-zhimeizhai`。  
- **包结构（若保留致美斋在核心一段时间）**：

```
com.miduo.cloud.infrastructure/
├── persistence/mybatis/po/       # 仅共用表 PO（如 OperateLogPO）
├── persistence/mybatis/mapper/
├── repository/                   # 接口实现
└── config/                      # 数据源等（可被变体覆盖）
```

- 石湾的 MySQL 表、PO、Mapper **全部在** `miduo-variant-shiwan` 内，不放入核心 infrastructure。

### 3.5 miduo-application（核心：通用服务 + SPI 接口）

- **职责**：  
  - 所有版本都要用的：操作日志、Bloom Filter 初始化、设备连接管理、异步配置等。  
  - 上传、采集、任务等：核心定义**接口**（如 `UploadProtocol`、`CollectPipeline`），默认实现或产线实现放在扩展/变体。
- **建议包结构**：

```
com.miduo.cloud.application/
├── log/                          # 操作日志（通用）
├── config/                       # 异步、Bloom 等（通用）
├── device/                       # 设备应用服务（通用）
├── spi/                          # 【新增】扩展点接口
│   ├── UploadProtocol.java       # 上传协议 SPI
│   ├── CollectPipeline.java      # 采集流水线 SPI（可选）
│   └── TaskLoader.java           # 任务加载 SPI（可选）
└── （致美斋专属：CodeApplicationService、TaskApplicationService 等迁到 miduo-variant-zhimeizhai）
```

- **复用方式**：各变体依赖 core，实现 SPI 接口并在本变体配置中注册。

### 3.6 miduo-controller（核心：通用 REST）

- **职责**：仅保留所有版本都有的 REST（如操作日志、设备配置、健康检查等）。  
- 产线专属接口（码采集、任务、上传等）迁到对应变体模块的 controller 包，或通过「核心定义接口 + 变体实现并注册」暴露。

### 3.7 miduo-frontend（核心：共用页面与框架)

- **职责**：License 相关 FXML/Controller、设备配置弹窗、操作日志、通用 CSS、HttpUtil、Stage 工具等。  
- **建议包结构**：

```
com.miduo.cloud.frontend/
├── controller/                   # 共用页面 Controller
│   ├── license/                  # License 系列
│   ├── IoDeviceDialogController.java
│   └── OperateLogController.java
├── service/                      # 设备连接、License 校验等（通用）
├── util/                         # HttpUtil、SpringContextUtil 等
└── resources/
    ├── fxml/                     # 共用 FXML（license/*.fxml、IoDeviceDialog.fxml、OperateLog.fxml）
    └── css/                      # 通用样式
```

- 主界面、任务管理、码查询/替换、数据上传等**产线形态差异大**的页面，放入对应变体，不放在核心 frontend。

### 3.8 miduo-bootstrap（核心：启动与配置入口）

- **职责**：  
  - 通用 Spring Boot 配置、全局异常处理。  
  - **按软件版本 ID（profile）** 加载对应变体的 Main 类与配置文件，不写死产线名。
- **建议包结构**：

```
com.miduo.cloud/
├── MiduoBackendApplication.java     # 通用 Spring Boot 入口（或仅作父类）
├── config/                          # 全局异常、MyBatis 等通用配置
└── variant/                         # 【新增】各变体启动入口（按 profile 选用）
    ├── zhimeizhai/
    │   ├── ZhimeizhaiLauncher.java      # 致美斋 JavaFX 入口
    │   └── ZhimeizhaiBackendApplication.java
    ├── shiwan/
    │   ├── ShiwanM1Launcher.java        # 石湾 1 号机
    │   ├── ShiwanM2Launcher.java        # 石湾 2 号机
    │   └── ShiwanBackendApplication.java # 或 M1/M2 各一个
```

- **复用方式**：通过 `config/variants/<id>.yml` 的 `build.mainClass` 或 Maven profile 指定用哪个 Launcher，实现「同一仓库、多入口、各产线独立包」。

## 四、变体层：各产线独立开发内容

每个**软件版本 ID** 对应一个变体模块（或一个变体模块内多个子模块，如石湾 m1/m2），**仅**放该产线专属实现与资源。

### 4.1 miduo-variant-zhimeizhai（致美斋）

- **对应**：软件版本 ID = `zhimeizhai`，配置文件 `config/variants/zhimeizhai.yml`。
- **建议结构**：

```
miduo-variant-zhimeizhai/
├── pom.xml
└── src/main/
    ├── java/com/miduo/cloud/variant/zhimeizhai/
    │   ├── application/              # 致美斋应用服务
    │   │   ├── CodeApplicationService.java
    │   │   ├── TaskApplicationService.java
    │   │   └── WmsUploadApplicationService.java（或放在 extension）
    │   ├── controller/                # 致美斋 REST（码、任务、上传等）
    │   ├── persistence/              # 致美斋 SQL Server PO/Mapper（若从 core 迁出）
    │   └── frontend/
    │       ├── MainController.java
    │       ├── TaskManagementController.java
    │       ├── CodeQueryController.java
    │       ├── CodeReplaceController.java
    │       ├── DataUploadController.java
    │       └── …
    └── resources/
        ├── fxml/                      # 致美斋主界面、任务、码查询等 FXML
        └── application-zhimeizhai.properties
```

- **依赖**：仅依赖核心模块（common、entity、domain、infrastructure、application、controller、frontend、bootstrap）；不依赖其他变体。

### 4.2 miduo-variant-shiwan（石湾）

- **对应**：软件版本 ID = `shiwan-m1` / `shiwan-m2`，配置 `config/variants/shiwan-m1.yml`、`shiwan-m2.yml`。  
- 可在一个变体模块内用包区分子机（m1/m2），或拆成 `miduo-variant-shiwan-m1`、`miduo-variant-shiwan-m2`（两个安装包时后者更清晰）。
- **建议结构**（单变体、内含 m1/m2）：

```
miduo-variant-shiwan/
├── pom.xml
└── src/main/
    ├── java/com/miduo/cloud/variant/shiwan/
    │   ├── application/              # 石湾应用服务（码包、四级关联、虚拟垛、上传等）
    │   ├── controller/                # 石湾 REST
    │   ├── persistence/              # 石湾 MySQL PO/Mapper（sw_* 表）
    │   ├── frontend/
    │   │   ├── machine1/             # 1 号机界面 Controller
    │   │   └── machine2/             # 2 号机 8 Tab Controller
    │   └── bootstrap/                # 可选：M1/M2 的 Launcher 若放变体
    └── resources/
        ├── fxml/
        │   ├── machine1/MainWindow.fxml
        │   └── machine2/*.fxml
        ├── application-shiwan-m1.properties
        └── application-shiwan-m2.properties
```

- **依赖**：核心 + 可选扩展；不依赖 `miduo-variant-zhimeizhai`。

## 五、版本配置与构建（与 04 文档一致）

### 5.1 config/variants 结构

```
config/variants/
├── variant-common.yml      # 全版本默认（端口、日志、通用 features）
├── zhimeizhai.yml
├── shiwan-m1.yml
└── shiwan-m2.yml
```

### 5.2 variant-common.yml（全版本默认）

```yaml
app:
  name: 赋码采集关联系统
  server:
    port: 8080

build:
  extensions: []
  variant: null

features:
  taskManagement: true
  codeCollect: true
  dataUpload: true
  codeQuery: true
  codeReplace: true
  operateLog: true
```

### 5.3 单版本配置示例：zhimeizhai.yml

```yaml
app:
  name: 赋码采集关联系统-致美斋

build:
  variant: zhimeizhai

# 可选：覆盖端口、日志等
```

### 5.4 单版本配置示例：shiwan-m1.yml / shiwan-m2.yml

```yaml
# shiwan-m1.yml
app:
  name: 赋码采集关联系统-石湾1号机

build:
  variant: shiwan
  machine: 1

# 数据库、G/N/M 等由 application-shiwan-m1.properties 提供
```

### 5.5 构建方式（Maven Profile）

- 父 `pom.xml` 中为每个变体定义 profile，仅在该 profile 下引入对应变体模块并指定 main class。  
- 与 04 文档一致：**输入** = 软件版本 ID + 发布版本号；**过程** = 读取 `config/variants/<id>.yml`，激活 profile，执行 `mvn clean package -Pvariant-zhimeizhai -Drelease.version=1.2.0`；**产出** = 该版本的 jar/安装包，命名含软件版本 ID 与版本号。

```xml
<!-- 父 pom.xml 片段 -->
<profiles>
  <profile>
    <id>variant-zhimeizhai</id>
    <modules>
      <module>miduo-common</module>
      <!-- ... 所有核心模块 ... -->
      <module>miduo-variant-zhimeizhai</module>
    </modules>
    <properties>
      <app.variant>zhimeizhai</app.variant>
      <main.class>com.miduo.cloud.variant.zhimeizhai.ZhimeizhaiLauncher</main.class>
    </properties>
  </profile>
  <profile>
    <id>variant-shiwan-m1</id>
    ...
  </profile>
  <profile>
    <id>variant-shiwan-m2</id>
    ...
  </profile>
</profiles>
```

- `miduo-bootstrap` 的打包 plugin 使用 `${main.class}`，即可实现「同一仓库、多入口、各产线独立包」。

## 六、复用代码清单与迁移顺序（实施参考）

| 当前位置 | 归属建议 | 迁移目标 |
|----------|----------|----------|
| miduo-common/* | 核心 | 不动 |
| ApiResult、PageInput/Output、IniFileUtil | 核心 | 已在 common，不动 |
| OperateLog 相关（entity + infra + application + controller） | 核心 | 保持或仅做包整理 |
| License 相关（entity + frontend service + FXML） | 核心 | 保留在 core frontend |
| DeviceConnectionManager、Tcp/Udp/Serial/Barcode 服务 | 核心 | 保留在 core frontend |
| BloomFilterManager、BloomFilterInitListener | 核心 | 保留在 core application |
| IoDevice 配置（PO + Mapper + Controller + Dialog） | 核心 | 保留在 core |
| CodeApplicationService、TaskApplicationService、WmsUploadApplicationService | 变体 | 迁至 miduo-variant-zhimeizhai |
| CodeRelationPO、ProductionOrderPO 等 SQL Server 表 | 变体 | 迁至 miduo-variant-zhimeizhai（推荐）或暂留 core |
| MainController、TaskManagement、CodeQuery、CodeReplace、DataUpload 等 | 变体 | 迁至 miduo-variant-zhimeizhai |
| 石湾专属（码包、四级关联、MySQL、双机） | 变体 | 全新写在 miduo-variant-shiwan |

## 七、扩展层（可选）

- 若某能力**多条产线需要、但不是全部**，可新增 `miduo-extension-<能力名>`，在 `config/variants/<id>.yml` 的 `build.extensions` 中声明。  
- 核心通过接口/SPI 调用扩展，不直接依赖 extension 模块；变体依赖需要的 extension。  
- 示例：`miduo-extension-report`（高级报表）、`miduo-extension-protocol-wms`（WMS 上传协议）。  

当前若没有明确「多产线可选」能力，可先不建扩展模块，后续按需增加。

## 八、小结

- **复用代码**：全部收敛在**核心层**（及可选**扩展层**），通过「核心不依赖扩展/变体 + 接口/SPI」保证一次修改、多版本受益。  
- **各产线独立开发**：每条产线对应一个 **miduo-variant-&lt;id&gt;** 和 **config/variants/&lt;id&gt;.yml**，独立迭代、独立构建、独立发布；核心与变体边界清晰，符合三份多版本管理文档的约定。  
- **实施顺序建议**：先建 `config/variants` 与 `miduo-variant-zhimeizhai`，把致美斋专属代码迁入变体并跑通 `-Pvariant-zhimeizhai` 构建；再建 `miduo-variant-shiwan` 与 shiwan-m1/m2 配置，实现石湾双机独立包；最后按需抽取扩展层并完善 feature-matrix。
