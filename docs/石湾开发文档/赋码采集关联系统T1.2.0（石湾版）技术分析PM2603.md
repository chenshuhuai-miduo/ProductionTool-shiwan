# 一、项目简介
## 项目背景
<font style="color:rgb(15, 17, 21);">当前石湾产线配备了两台工控机、五台采集相机及相关剔除装置和报警灯，用于瓶码、盒码和箱码的采集与关联。其中1号机通过两台瓶码相机交叉去重采集瓶数据，一台盒码相机采集盒数据；2号机接收1号机数据，并通过一台盒码相机和一台箱码相机完成盒箱关联与垛码绑定。整个流程依赖一套由第三方开发的赋码采集关联系统进行控制。</font>

<font style="color:rgb(15, 17, 21);">然而，该第三方系统在日常运行中问题频发，几乎每天都会出现大小故障，严重时甚至需要手动操作数据库进行数据修正，直接影响了产品的入库效率和出货准确性。由于技术依赖，供应商的维护响应滞后，需求变更无法及时落实，不仅增加了产线的运维成本，也对公司品牌形象造成负面影响。</font>

<font style="color:rgb(15, 17, 21);">在功能层面，现有系统缺乏码包导入与管理、码关系比对、操作帮助提示等关键能力，导致数据准确性难以保障，且米多作为兜底方需人工校验码包关系。此外，产线重要参数及设置页面缺少密码保护，操作人员易因误操作更改配置，引发生产中断。</font>

<font style="color:rgb(15, 17, 21);">上述问题均源自石湾产线日常使用中的实际反馈与操作人员的使用体验。为从根本上解决技术依赖、功能缺失、数据质量低下及操作安全隐患，亟需自研一套全新的赋码采集关联系统，以实现技术自主可控，提升系统稳定性和功能完整性，保障产线高效可靠运行。</font>

## 项目目的
<font style="color:rgb(15, 17, 21);">本项目旨在自主研发一套全新的赋码采集关联系统，以替代现有的第三方软件，实现技术自主可控，降低维护成本与响应延迟。在保留旧版核心功能的基础上，系统将新增码包管理、码关系比对、自动上传等功能，增强数据管理能力与追溯准确性；同时通过优化操作界面、增加帮助提示和密码保护等措施，降低误操作风险，提升用户体验。最终，提供稳定可靠的系统，减少因软件故障导致的生产中断和数据错误，及时响应需求变化，改善客户使用体验，从而维护和提升公司品牌价值。</font>

### 2.瓶盒关联软件
**业务流程：瓶 1/瓶 2 双相机读码 → 交叉去重 → 瓶码校验（码包范围、重码、错码）→ 满 G 瓶后采盒码 → 盒码校验 → 写 coderelationupload（产品、单号来自当前任务表）+ 若有异常则写待剔除盒表并写日志。**

#### 2.1.部署与数据源
1 号机：不安装 MySQL 服务，应用通过 JDBC 连接 2 号机本机 的 MySQL（spring.datasource.url=jdbc:mysql://<2号机IP>:3306/...）。  
2 号机：本机部署 MySQL，与 1 号机共用同一库（同一 schema），1 号机写入即 2 号机可读，无需额外同步。  
网络：1 号机与 2 号机在同一局域网，2 号机防火墙/MySQL 允许 1 号机 IP 访问 3306

#### 2.2.1号机进行瓶盒关联
2 号机：在“选择/切换当前生产任务”时，将当前订单号、产品编码（及业务需要的产品名等）写入共用库的当前任务表ProductionOrder。  
1 号机：在每次要写入 coderelationupload 前，先查当前任务表，取到当前订单号、产品编码后再写入瓶盒关联记录（含产品、单号字段）。若当前任务表无有效记录（2 号机未选单），1 号机不写入 coderelationupload，并在界面提示“请先在 2 号机选择生产任务”。

#### 2.3.码包与校验（1 号机只读 2 号机库）
码包在 2 号机 导入（P02-03），数据在共用库的码包热表（及冷表）中。  
1 号机：通过 JDBC 查同一库的码包热表，做瓶码（盖外码小标）、盒码（盒外码中标）的“在码包范围内”校验。

#### 2.4.剔除：1 号机“待剔除”与 2 号机“剔除记录表”
1 号机：

对重码/错码/超码/缺码等：不建立瓶盒关联、不计数、写日志，并将“该盒待剔除”写入共用库，供 2 号机在盒箱垛环节读取。<font style="color:rgba(20, 20, 20, 0.92);background-color:rgb(252, 252, 252);">将CodeRelationUpload表对应数据的status字段为3。</font>  
2 号机：

在执行剔除时，往您提供的剔除记录表写入一条记录，字段包括：箱码、盒码、瓶码明细、问题码、剔除原因、剔除时间、生产单号等。

#### 2.5.报警规则
| 触发条件 | 报警类型 |
| --- | --- |
| 设备连接状态：检测未连接 | 长报警 |
| 重码/错码/超码/缺码等异常 | 短报警 |
| 连续重码 3 次 | 长报警 |


将1号机出现重码的次数记在内存中，达到3次后触发长报警；关闭报警后将内存中的重码数量清空

#### 2.6. 配置与工程结构
1 号机配置：  
数据源：2 号机 IP、端口、库名、用户名、密码；提供“连接测试”（连 2 号机 MySQL）。  
系统设置中：页面配置仅“数据采集”Tab；不配置虚拟垛标、上传（1 号机用不到）。  
代码：石湾 1 号机只复用公共层（设备、DB 访问、工具类等），瓶盒关联业务与 2 号机盒箱垛关联分开实现；1 号机不包含码包导入、上传、垛相关逻辑。

##### 3.2 当前任务与 1 号机协同
+ **选单后写入**：用户在 2 号机选择产品并填写/确认生产单号，在点击「开始采集」且门禁通过（含二次确认弹窗）后，将**当前订单号、产品编码（及业务需要的产品名等）****写入共用库****当前任务表**（表名与 1 号机方案一致），并在此后采集中使用同一任务信息。
+ **开始采集门禁**：码包已导入（至少小标、中标、大标满足当前生产）、设备检测通过（相机、剔除装置等），再弹出二次确认（生产信息 + 包装比例 1 垛 X 箱、1 箱 X 盒），确认后开始采集并写入当前任务。

##### 3.3 盒箱垛采集流程（环节 4～6）
+ **数据来源**：从共用库**实时读取** 1 号机写入的瓶盒关联数据（同一 DB，无需同步）；盒码、箱码由 2 号机本机相机采集（龙门架：盒码相机 + 箱码相机）。
+ **环节 4 - 箱码关联**：
    - 每 N 盒（如 4 盒）组成一箱时，采集箱码；**先查码关系表**：若该箱内任一带 1 号机剔除标记的盒，则整箱不计数、不建关联、写日志、**执行剔除**（调用剔除装置）、**写入剔除记录表**（箱码、盒码、瓶码明细、问题码、剔除原因、剔除时间、生产单号）。
    - 若未标记待剔除：校验盒箱关系（N 盒与箱码一一对应）、箱码在箱外码大标码包内、非重码；通过则建立盒-箱关联、箱码计数+1；不通过则同样不计数、不建关联、写日志、执行剔除、写剔除记录表。
+ **环节 5 - 垛码关联**：箱码有效计数达到 M（如 70 箱）时，生成**虚拟垛标**（格式：前缀+年月日+序号+产线号+随机数，配置在系统设置→业务→虚拟垛标规则），建立 M 箱与垛码关联；失败则程序内重试。
+ **环节 6 - 数据上传**：成垛完成后，若「系统设置→业务→上传配置」中自动上传开关为开，则**自动上传**该垛至米多；成功则绿灯常亮约 1 分钟后熄灭，失败则红灯+蜂鸣，报警信息区展示具体失败原因（到箱/盒/瓶），需人工关闭报警并可到 P02-08 手动重传。**调取接口api/sign/md.fc.Store/v1/SyncCodeAndVirtualRelation上传。由于该方法时异步执行的，需轮询查询数据上传状态，调取接口api/sign/md.fc.Store/v1/GetSyncCodeAndVirtualRelationResult。**

##### 3.4 设备与报警
+ **设备**：盒码相机、箱码相机、剔除装置、报警灯/蜂鸣器；参数与连接在「系统设置」中配置（与 P03-02 共用）。
+ **报警**：连续重码 3 次、上传异常为**长报警**，需人工点击「关闭报警」；其余可为短报警。关闭报警时向报警设备发停止指令（协议按设备/Modbus 等配置）。

##### 3.5 其他交互
+ **强制满垛**：未满 M 箱时也可结束当前垛，生成虚拟垛标并重置箱数计数。
+ **提取工单未成垛**：输入/扫箱码 → 查订单 → 确认 → 回显主界面继续生产（尾数板续产）。
+ **收回剔除**：剔除装置动作后若未自动收回，通过「收回剔除」发 PLC 收回指令（设备在系统设置中配置）。
+ **剔除数清零**：仅界面将「总剔除数」显示清零，不删剔除记录表数据。
+ **无需采集码**：开关打开时，读码设备不工作、不采集任何数据（满足「当天不生产五码合一」场景）。

#### 3.4.P02-03 码包管理
+ **码包类型**：盖外码小标、盒外码中标、箱外码大标；导入后写入**热表**（未使用），采集/关联时查热表，关联成功后落冷。
+ **在线更新**：调取开放平台接口/api/v1/code/package/query-completed，仅**盖外码小标、箱外码大标**；统一 URL + type 参数；拿到链接后解析并下载文件到本地，再从本地文件读取写入数据库。按类型**分别**记录上次成功拉取时间，增量拉取；启动时自动执行一次；某类型失败则不更新该类型时间戳、不覆盖本地。
+ **本地导入**：三种类型均可；**盒外码中标仅能本地导入**。先选码包类型 → 选 TXT 文件 → 输入密码（123456）→ 解析入库；重复判断按**文件名+内容**（热表+冷表或 all_codes 视图），避免重复/高度重叠入库。**导入时如何快速判断重码？**
+ **列表与操作**：表格列（类型、名称、导入时间、导入方式、码数量去重、状态、备注、操作）；操作：**查看**（弹窗展示该包所有码，支持搜索与分页，展示码值、关联状态、关联时间）、**删除**（仅当该包内无任何已关联码时可删，逻辑删除）。
+ **删除与使用完**：逻辑删除；码包内码全部落冷后仍保留记录，不可删除（供码替换等使用）。

#### 3.5.P02-04 数据查询
+ **查询**：输入任意一层码（瓶/盒/箱/垛）→ 查该码所属**垛**的完整数据（垛-箱-盒-瓶），左侧分层表格（第一层瓶码、第二层盒码、第三层箱码、第四层垛码），输入码对应单元格**红色**；右侧选中行详情（码、采集时间、产品编号、产品名称、生产单号）。
+ **实现**：按码反查所属垛，再按垛查全量关联行，按层级分列展示；支持手动输入与扫码枪、回车触发查询。

#### 3.6.P02-05 数据替换
+ **范围**：仅外码（盖外码/瓶码、盒码、箱外码），不替换内码。
+ **规则**：原码必须在系统中存在；新码须与原码**同层级**、在对应码包**热表**内、未使用、非原码。确认弹窗+密码 123456。
+ **执行顺序**：先查该码关联是否已上传，即在CodeRelationUpload表内字段Status为1的；**未上传**则校验通过后直接改本地；**已上传**则先调**云端替换接口**成功后再改本地；若码类型有内码（如盖、箱），需**同步替换云端内码**；确认弹窗中需说明「已上传先云端后本地」「有内码会同步云端内码」。

#### 3.7.P02-06 取消关联
+ **单码模式**：输入瓶码或盒码 → **子查父**得到上级单元 → 取消该**上级单元**及其下全部（扫瓶取消盒，扫盒取消箱）；**不支持**箱码、垛码输入（防误操作整垛取消）。
+ **多码模式**：输入盒/箱/垛码加入列表（可多个、可删）→ 识别 → 按**垛→箱→盒**顺序依次取消；若盒属于某箱，先取消箱时已连带取消盒，后续自动跳过。
+ **通用**：取消前查**是否已上传**；已上传则**先取消云端再取消本地**；取消后码**回迁**（冷表→热表）；二次确认+密码 123456；有上级未取消时禁用确认并提示先取消上级。

#### 3.8.P02-07 生产统计
+ **生产统计**：按日期范围（可选生产单号）统计**垛数、箱数、盒数、剔除数**；垛数、剔除数**可点击**。
    - **点击垛数**：弹窗「垛码列表」表格（垛码、箱数、生产单号、关联时间），支持按垛码筛选、重置。
    - **点击剔除数**：弹窗「剔除记录」：数据来自**剔除记录表**；左侧按瓶-盒-箱分层表格，问题码红色；右侧选中行详情（问题码、箱码、剔除原因、剔除时间、生产单号）；支持按箱码筛选、重置。
+ **上传统计**：按日期、生产单号、状态（全部/成功/异常）查上传记录；表格列：垛码、箱数、生产单号、上传时间、状态、异常原因（具体到箱/盒/瓶）；分页、默认按上传时间倒序。

#### 3.9.P02-08 数据上传
+ **自动上传**：成垛后若开关开启则自动上传；根据上传结果在码上传记录表中进行更新，并在**上传信息展示区**推送日志（开始上传→上传中→成功/失败），格式与文档一致；失败原因展示到箱/盒/瓶（依赖服务端返回 details，见下）。
+ **手动上传**：批量上传所有未上传垛；日志同样进入上传信息展示区。
+ **垛码状态**：输入垛码可**查询状态**（成功/失败/未上传）、**设为未上传**（用于修数后重传）、**设为已上传**（避免重复上传）；确认提示与文档一致。
+ **服务端接口**：失败时建议返回 **details** 数组（层级、问题码、原因），客户端拼成可读文案用于报警区、P02-08、P02-07 异常原因列；成功可仅返回成功标识。
+ **调取接口api/sign/md.fc.Store/v1/SyncCodeAndVirtualRelation上传。由于该方法时异步执行的，需轮询查询数据上传状态，调取接口api/sign/md.fc.Store/v1/GetSyncCodeAndVirtualRelationResult。**
